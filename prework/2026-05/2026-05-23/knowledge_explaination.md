# 2026-05-23 知识讲解

## 今日知识主线

从文件中契约到系统分层，再到业务规则的不变性——今天的概念串起来看，其实都在回答同一个问题：**怎么让多个模块、多个Agent、多条流水线在配合的时候不吃错文件、不越界操作、不破坏业务底线？** 我们按从易到难的顺序，讲三个新概念：`输入契约`、`分层架构`、`领域不变量`。

---

## 概念一：输入契约

- **领域**：软件工程
- **难度**：★★☆☆☆

### 亲和完整讲解

想象一下，你和朋友接力做一道菜：你负责洗菜切菜，朋友负责炒。如果你只把切好的菜放在厨台上，没有用盘子装，也没有告诉朋友“菜在左台面”，朋友可能找不到，或者以为你没做完。你们之间其实缺少一个**契约**：约定好菜放在哪里、用什么容器、必须是什么状态（切好、洗好）。

在软件系统里，“输入契约”就是下游任务对上游产物必须满足的硬性约定。今天的 `DailyLearningAssistant` 仓库把每日学习自动化的输入范围从5个仓库扩展到6个（加入了 `mcp`），随之而来的就是更新**输入契约**：阶段2（生成概念文件）必须拿到6个 `work_summary_*.md` 文件，而且每个文件都要存在、非空、可读取。如果少一个、文件是空的、或者权限不对，下游就不能开始工作。

为什么不放宽松一点？因为下游一旦基于坏材料生成了内容，问题就会悄悄蔓延——你以为有6条信息，其实只读了5条，生成的日报就会漏掉 `mcp` 的知识点。这就是“输入契约”存在的理由：**在入口处把问题拦住，不让它流到后面去**。

### 有趣事例

你可以把它想象成机场安检。登机口必须确认每位乘客都有登机牌、身份证、且本人到场，才会放行。如果安检员看到一个人没有登机牌就让他登机，可能导致飞机上多了一个不该上的人。输入契约就是自动化流水线里的安检员。

### 常见误区

- **误区**：“只要文件存在就行，空文件就跳过。” — 空文件可能导致下游解析出错，或者误以为数据有效。契约里要明确要求“非空”。
- **误区**：“契约写文档里就行，代码里可检查可不检查。” — 契约必须落在代码里自动执行，否则只是“约定”，不是“契约”。

### 与前一天工作内容的关系

前一天（5月22日）虽无记录，但更早的5月21日讲过“上游产物完整性校验”，这和输入契约非常像。区别在于：完整性校验是一种检查动作，而输入契约是**检查什么**的约定内容。今天我们把“放入契约”这个概念单独提炼出来，强调它作为阶段间的稳定协议。

### 一个具体例子

在 `DailyLearningAssistant` 的多阶段流水线中，阶段2的脚本会先读取一个配置文件，里面写明需要检查的文件列表（`work_summary_*.md`）。然后它遍历这个列表，对每个文件调用 `exists()`、`readable()`、`nonEmpty()` 三个判断。全部通过才继续生成概念文件。这就是一次典型的输入契约落地。

### 与其他概念的关联

- **前置完整性校验**：输入契约的具体检查逻辑。
- **数据谱系**：输入契约保证了每份知识日报的来源都是完整的，这样数据谱系才能可信。
- **文档与执行逻辑同步**：当仓库数量变化时，输入契约的配置文件必须同步更新，否则校验规则和实际需求会脱节。

### 可继续深入学习的问题

1. 如果输入契约的检查失败，应该让流水线完全停止，还是走降级路径？
2. 文件格式（比如必须是 Markdown）要不要也放进契约？怎么校验？
3. 跨流水线的输入契约怎么设计？比如一个流水线的输出是另一个流水线的输入。

## 概念二：分层架构

- **领域**：软件工程
- **难度**：★★★★☆

### 亲和完整讲解

假设你要建一个图书馆。你可以把所有东西都堆在一个大房间里：借书还书台、书架、管理员休息室、电脑系统全混在一起。但那样会非常混乱：读者找不到书，管理员找不到设备，出了故障也不知道修哪里。

更好的做法是把图书馆分成不同区域：一楼是借还区，二楼是阅览区，地下室是书库，每个区域有明确职责。这就是“分层架构”的思想：**按职责把系统划分成若干层，每层只做自己该做的事，并且通过稳定的接口与上下层协作。**

今天 `ResearchPaperBase_codex` 仓库把论文系统设计文档按6层重组：

1. **UI（用户界面）** — 只负责展示和输入收集，不做业务判断。
2. **API Boundary（API边界）** — 把用户操作映射成命令和查询，做鉴权和错误信封。
3. **Application Use Cases（应用用例）** — 编排业务流程，比如“提交一篇论文待审”。
4. **Domain Core（领域核心）** — 表达业务世界里永远成立的规则和对象（比如“论文必须有标题”）。
5. **Agent & Knowledge Services（Agent与知识服务）** — 放Graph-RAG、知识版本管理等智能能力。
6. **Infrastructure（基础设施）** — 数据库、消息队列等。

每一层只能依赖它下面的层，不能跨层调用。比如UI不能直接操作数据库，必须通过API→用例→领域→基础设施的顺序走。这样好处是：如果将来要换数据库，只需要改最底层，上面的逻辑完全不受影响。

### 有趣事例

想象一个餐厅：服务员（UI）只负责点单上菜，不负责炒菜；厨师（用例）只负责按菜单做菜，不负责洗碗；采购（基础设施）只负责买食材。如果今天厨房的炉子换了，服务员和厨师的工作方式都不需要变，因为炉子是基础设施层的东西。

### 常见误区

- **误区**：“分层就是文件夹分层。” — 真正的分层不仅是目录结构，更重要的是**依赖方向**：上层可以调用下层，下层绝对不能反过来调用上层。如果上层需要从下层得到什么，必须通过接口回调或事件通知。
- **误区**：“层越多越好。” — 每增加一层都会增加理解和维护成本，分层应该是为了隔离变化，不是为了分而分。

### 与前一天工作内容的关系

今天的分层架构是 `ResearchPaperBase_codex` 的主要贡献，它没有直接出现在之前的知识讲解中。但可以与5月13日讲过的“契约式开发”联系起来：分层架构本身就是通过定义每层的“契约”来约束系统行为。

### 一个具体例子

`ResearchPaperBase_codex` 中的 `ProjectPermission` 权限模型放在Domain Core层。假设未来要增加OAuth登录方式，那UI层和API Boundary层要改，但Domain Core里的权限判断逻辑（比如“Owner不能失去自己Project的全部权限”）完全不用动。因为业务规则是稳定的，接口协议是易变的。

### 与其他概念的关联

- **API Boundary / Contracts**：定义层与层之间的接口协议，是分层架构落地的关键。
- **Domain Core**：分层架构中最核心的一层，存放领域不变量。
- **Application Use Cases**：连接外界请求和领域核心的编排层。
- **内容保护**：分层架构帮助把敏感内容（如密钥、知识版本）隔离在正确的层中，防止UI层误操作。

2. 如何判断一个功能属于Domain Core还是Application Use Cases？

## 概念三：领域不变量

- **领域**：软件工程 / 计算机科学
- **难度**：★★★★☆

### 亲和完整讲解

“不变量”这个词听起来挺数学的，其实很简单：就是**无论系统怎么变、什么时候执行，都必须永远成立的规则**。比如在一个图书馆系统里，“每个读者最多借10本书”就是一个不变量——不管借书流程怎么改、数据库怎么换，这个数字都不能被打破。

在软件开发里，“领域不变量”特指业务世界本身固有的规则。今天 `ResearchPaperBase_codex` 中特别强调了一个不变量：**Owner默认全权限不变量**。意思是：Project的创建者（Owner）对自己的Project必须拥有全部权限（`access`、`use`、`delete`），这个权限不能被其他用户撤销，也不能被Owner自己放弃。为什么？因为如果Owner连删除自己Project的权限都没有，那这个Project就变成了“孤儿”，谁也删不掉，系统就会出现无法清理的资源。

而且领域不变量必须放在Domain Core层，不能放在数据库里，也不能放在API层。理由是：数据库可能被直接操作绕过业务逻辑，API层可能被篡改。只有放在Domain Core，才能保证任何入口进入的操作都会经过这个约束。

### 有趣事例

想想足球比赛：球门的大小是固定的，不管比赛在哪个球场、天气好不好，门高2.44米、宽7.32米这个规则不能变。这就是足球场上的“领域不变量”。如果随便改球门大小，比赛就无法公平了。

### 常见误区

- **误区**：“不变量写在数据库约束里就足够了。” — 数据库约束只能管数据一致性，但领域不变量还涉及状态机、事件、权限等更复杂的规则。数据库无法阻止一个第三方应用用错误的逻辑绕过程序。
- **误区**：“不变量永远不变。” — 不变量是当前业务版本的规则，如果业务需求演进，不变量可以被修改。但修改需要非常谨慎，且必须同步更新所有依赖它的地方。

### 与前一天工作内容的关系

今天与5月13日讲过的“状态机与并发锁”有相似之处：状态机里的状态转移也是一种不变量（比如“已提交”不能直接回到“草稿”）。而领域不变量更强调业务语义。

### 一个具体例子

在 `ResearchPaperBase_codex` 的 `ProjectPermission` 模型中，有一个检查方法：
```
if user == project.owner:
    assert user.hasAllPermissions(project)
```
这个断言放在Domain Core的每个修改权限的方法开头。无论你是通过API、命令行还是后台脚本去改权限，只要Owner的权限被修改了，这个断言就会触发并阻止。这就是不变量在代码层面的保护。

### 与其他概念的关联

- **Domain Core**：领域不变量是Domain Core的核心组成部分。
- **ProjectPermission 权限模型**：Owner不变量是这个模型的安全底线。
- **内容保护**：不变量也是内容保护的一种手段，防止恶意修改。
- **输入契约**：两个概念都是“契约”在不同层面的体现——输入契约是文件级契约，领域不变量是业务级契约。

## 整体关联总结

今天的三个概念像一座金字塔：

- **底部（输入契约）**：给多阶段自动化流水线的阶段间建立稳定的文件交接协议，保证下游接收的是完整、合格的输入。
- **中部（分层架构）**：给整个系统按职责划分边界，让每一层各司其职，通过接口契约协作。
- **顶部（领域不变量）**：在分层架构的核心层（Domain Core）中，固定业务世界不可动摇的规则，保证无论系统如何演化，底线不被突破。

它们共同回答一个问题：**如何让复杂系统在多人、多Agent、多阶段协作时，不出错、不越界、不失信？** 答案就是用契约（输入契约、API契约、不变量）在各个边界上设立“检查点”。

---

## 适合放入学习日报的精简结论

- **输入契约**：阶段之间的文件约定（存在、非空、可读），让自动化流水线拒绝坏材料。
- **分层架构**：按职责（UI、API、用例、领域、智能、基础设施）分层，每层只做自己的事，依赖方向从上到下。
- **领域不变量**：业务中永远成立的规则（如Owner必须拥有全部权限），放在核心层用代码保证。

今天的三个概念都在说同一件事：**在系统边界上放稳协议，让不确定性变成可检查、可审计、可执行的规则。**
