### `API_design.md`

围绕 API 设计整理了 8 个角度：职责分离、系统理解、前后端混乱防治、业务逻辑保护、测试、AI 辅助开发安全、未来变更支持，以及数据形状定义。它把 API 从“接口地址列表”提升为前后端协作契约和业务边界保护层。

### `FR_design.md`

记录了功能需求文档的边界：FR 应描述用户可见能力、业务规则、权限边界、状态变化、失败路径和不可做事项；不应直接承担数据库字段、API 响应或前端 ViewModel 字段定义。该笔记特别强调 P0 FR 要能驱动 UI、状态机、数据、API、权限和 QA 的后续闭环设计。

### `ai_desicion_edge.md`

新增 AI 可参与事项的风险分级表。页面布局、React 组件、测试用例和文档可以更多交给 AI；数据库 schema、认证方案、权限模型、Agent 状态机、安全策略等则需要更高强度的人类审查。这为 AI 协作开发划出了“建议、可做、必须审查”的边界。

### `layer_design.md`

梳理需求、UI 信息架构、API ViewModel、数据层、权限安全层之间的职责。文档明确：FR 不应该解释“向前端暴露什么数据”，字段名、枚举、JSON schema、表结构、索引和约束应落到 API 或数据契约层。

## 主要工作主题

1. **从提示式开发转向契约式开发。** 前一天新增内容共同强调：AI 参与开发时，必须先有产品说明、架构说明、数据模型和 API 契约，否则模型很容易把需求、接口、数据和 UI 混在一起。
2. **把 FR 从技术字段中解放出来。** 笔记反复区分用户可见能力与后端实现细节，避免需求文档提前绑定字段名和 schema。
3. **定义 AI 决策边界。** AI 适合做布局、组件、测试、文档等中低风险工作；认证、权限、安全、状态机和 schema 属于高风险区域，需要人类把关。
4. **强调垂直切片和可审核 diff。** Web App 开发不应一次性铺开全部功能，而应围绕一条完整功能链推进，每次都能被测试和审核。
