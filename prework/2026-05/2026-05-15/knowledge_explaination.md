# 2026-05-15 知识点讲解

## 今日知识主线

今天的知识主线可以概括为一句话：**从大语言模型的一次生成，到 Agent 的上下文控制，再到软件需求的可验收表达，本质上都在回答同一个问题：复杂系统怎样把“不确定”变成“可计算、可管理、可验证”的过程。**

当天材料里，一条线来自 `interview_prepare`：它把 LLM 架构讲成从文本、分词、嵌入、位置编码、Transformer Block、LM Head 到概率输出的完整流程。另一条线来自 `AInote` 和 `DailyLearningAssistant`：Agent 自动化任务需要管理上下文、逐步披露信息、分类失败并在发布前做校验。第三条线来自 `ResearchPaperBase_codex`：需求文档需要写清验收标准、权限边界、软删除、幂等和私人资产清理。

今天选择三个概念：

1. **Transformer 前向推理流程**：理解 LLM 如何把一段文字一步步变成下一个 token 的概率分布。
2. **上下文管理**：理解为什么 Agent 不能简单地“给越多材料越好”，而要控制输入范围、顺序和阶段。
3. **可测试需求**：理解好需求不只描述“要做什么”，还要说明怎样判断它真的做对了。

## 一、Transformer 前向推理流程

### 概念名称

**Transformer 前向推理流程**（Transformer Forward Inference Pipeline）

### 概念属于哪个领域

**计算机科学 / 深度学习 / 大语言模型**

它是理解现代大语言模型如何工作的核心入口，和 BPE 分词、Token Embedding、位置编码、Self-Attention、LM Head、自回归生成等概念紧密相连。

### 难度评级

★★★★（四星 / 较难）

它难在不是单个公式，而是一条连续的数据流水线：文本先变成 token，token 再变成向量，向量经过多层注意力和前馈网络，最后被映射成词表上的概率。只要能抓住“表示形式不断变化”这条主线，就会清楚很多。

### 概念讲解

同学，我们先把大语言模型想象成一条非常精密的翻译流水线。不过它翻译的不是中文到英文，而是把“人能读懂的文字”一步步翻译成“机器能计算的数字”，再把计算结果翻译回“下一个最可能出现的词”。

假设你输入一句话：

```text
人工智能正在改变
```

模型要做的事情不是一次性“理解整句话然后写作文”。更准确地说，它要回答一个局部问题：**在已有上下文后面，下一个 token 可能是什么？**

这个过程大致可以拆成七步。

**第一步：文本输入。**

人类输入的是文字，里面有汉字、英文、空格、标点。神经网络不能直接吃文字，它只能处理数字，所以必须先转换。

**第二步：BPE 分词。**

分词器会把文本切成 token。token 不一定等于一个字，也不一定等于一个完整词，它可以是常见片段、子词、标点或字节片段。BPE 的基本思想是：先从很小的单位开始，不断合并语料中最常见的相邻片段，最终形成一个既能表示常见词、又能覆盖陌生词的词表。

**第三步：Token Embedding。**

分词后，每个 token 会对应一个整数 id。但 id 本身没有语义，`100` 并不比 `99` “更聪明”。Embedding 层的作用就是查表：把每个 token id 换成一个稠密向量。这个向量像是 token 在一个高维语义空间里的坐标，语义或用法相近的 token 往往会在空间中更接近。

**第四步：加入位置信息。**

Transformer 的注意力计算本身很擅长看“谁和谁有关”，但它并不天然知道第一个 token、第二个 token、最后一个 token 的顺序差异。位置编码就是给向量加上“座位号”。传统正弦位置编码用不同频率的正弦和余弦函数表示位置；RoPE 则通过旋转向量的方式把相对位置信息注入注意力计算。这里基于当日材料可确认的是：位置编码的目的，是让模型在处理序列时拥有顺序感。

**第五步：经过 Transformer Block。**

这是模型的主体部分。每一层通常包含 Self-Attention 和前馈网络。Self-Attention 会让每个 token 根据 Query、Key、Value 去“观察”上下文中的其他 token，判断哪些信息最相关。比如在“苹果发布了新手机，它的芯片更快”里，“它”应该更多关注“新手机”而不是水果苹果。

前馈网络则像每个位置上的小型加工车间，对注意力整合后的信息进一步变换。很多层堆叠起来，模型就能从浅层的词形、局部搭配，逐步形成更抽象的语义、结构和推理线索。

**第六步：LM Head 输出 logits。**

Transformer 最后一层会给每个位置输出一个隐藏向量。LM Head 的任务是把这个隐藏向量映射到整个词表的维度。假设词表有 10 万个 token，LM Head 就会给这 10 万个候选 token 各自打一个分数，这些分数通常叫 logits。

**第七步：概率分布与自回归生成。**

logits 会通过 softmax 等方式转成概率分布。模型可以选择概率最高的 token，也可以按温度、top-k、top-p 等采样策略从高概率候选中抽取一个。选出的 token 会被接到原文本后面，再进入下一轮推理。这样一步一步生成，就叫自回归生成。

所以，Transformer 前向推理流程的核心逻辑是：

```text
文字 -> token id -> 向量 -> 带位置的向量 -> 上下文加工 -> 词表分数 -> 下一个 token 概率
```

这条链路最迷人的地方在于：模型看起来像是在“说话”，但底层其实是在重复做一个非常具体的计算任务：根据已有上下文，预测下一个 token 的概率分布。

### 有趣的相关事例

你可以把 LLM 想象成一位超级接龙选手。普通接龙只看最后一个词，LLM 则会看前面很长一段上下文，还会判断哪些词更重要。它不是简单背答案，而是在高维向量空间里综合语义、语法、风格和任务指令，然后决定下一个 token。

### 常见误区

**误区一：模型一次性生成整篇文章。**

实际生成通常是一步一步来的。每一步预测一个或一小段 token，再把结果放回上下文继续预测。

**误区二：token 就等于汉字或英文单词。**

token 是模型词表里的基本单位，可能是字、词、子词、标点或字节片段。分词方式会影响模型处理文本的效率和细粒度。

**误区三：Embedding 向量就是人类可直接解释的意义。**

Embedding 确实承载语义信息，但每个维度通常不是“情感”“颜色”“动作”这种直接标签。它更像大量统计关系压缩出来的坐标。

### 与前一天工作内容的关系

前一天讲过 Prompt 约束工程。今天的 Transformer 前向流程能解释为什么 prompt 会影响输出：prompt 进入模型后会变成上下文向量，参与注意力计算，最终改变下一个 token 的概率分布。换句话说，提示词不是“魔法咒语”，而是改变模型计算条件的输入数据。

### 一个具体例子

输入：

```text
今天我们学习 Transformer，它的核心机制是
```

模型先把这句话分成 token，再把 token 转成向量，加入位置信息，经过多层 Transformer Block。最后在当前位置，LM Head 给词表中每个候选 token 打分。

可能的高概率候选包括：

| 候选 token | 可能原因 |
| --- | --- |
| 注意力 | 前文出现 Transformer，常见搭配是“核心机制是注意力”。 |
| Self | 英文材料里常接 “Self-Attention”。 |
| 多头 | Transformer 常见术语是“多头注意力”。 |
| 编码器 | 如果上下文偏原始论文介绍，可能接编码器/解码器。 |

如果采样结果是“注意力”，下一轮输入就变成：

```text
今天我们学习 Transformer，它的核心机制是注意力
```

模型再继续预测后面的 token。

### 与其他概念的关联

| 关联概念 | 关系说明 |
| --- | --- |
| BPE 分词 | 前向流程的入口，把文字转成 token id。 |
| Token Embedding | 把离散 token id 变成神经网络可计算的向量。 |
| 位置编码与 RoPE | 给序列加入顺序信息，使注意力能利用位置关系。 |
| Self-Attention | Transformer Block 的核心，让 token 根据上下文重新组织表示。 |
| LM Head | 前向流程末端，把隐藏状态映射为词表分数。 |
| 自回归语言模型 | 通过反复执行前向推理，一步步生成文本。 |

1. 为什么 Self-Attention 可以并行处理训练序列，而自回归生成时仍然要一步步生成？
2. RoPE 为什么更适合表达相对位置关系？
3. 温度、top-k、top-p 这些采样参数怎样改变生成文本的稳定性和创造性？

## 二、上下文管理

### 概念名称

**上下文管理**（Context Management）

### 概念属于哪个领域

**大语言模型 / Agent 工作流工程 / 信息架构**

它既和 LLM 的上下文窗口、注意力分布有关，也和多 Agent 协作、渐进披露、Prompt Engineering 中的上下文裁剪有关。

### 难度评级

★★★★（四星 / 较难）

它表面上像是“材料放多少”的问题，实质上是“让模型在有限注意力和有限窗口里看到正确材料”的工程问题。难点在于取舍：少了会缺证据，多了会引入噪声。

### 概念讲解

同学，我们先想象你正在准备考试。老师给你一本重点讲义，你可能学得很快；如果老师把整个图书馆都搬到你桌上，并说“答案都在里面”，你反而会崩溃。

Agent 也是这样。

LLM 确实可以处理很长的上下文，但“能放进去”不等于“都能用好”。上下文越长，里面越可能混入旧信息、无关细节、相互冲突的要求和过期结论。模型在生成时会根据上下文预测后续内容，如果重要信息被埋得太深，或者噪声材料太多，输出就可能跑偏。

**上下文管理**就是对输入材料进行组织和控制：哪些内容要给模型，哪些暂时不给；先给什么，后给什么；哪些是硬约束，哪些只是参考；多 Agent 串行审阅时，每个 Agent 应该看到怎样的材料。

它的核心目标不是“让模型知道一切”，而是**让模型在当前任务中知道足够正确、足够相关、足够有优先级的信息**。

我们可以从四个层次理解它。

**第一层：容量管理。**

模型有上下文窗口限制。即使窗口很大，也不是无限。长文档、日志、代码库、历史对话都可能占满空间。容量管理要决定：当前任务最需要哪些片段？比如生成知识讲解时，最关键的是当天 `concept_relevance.md` 和月度知识记录，而不是所有历史日报正文。

**第二层：相关性管理。**

并不是所有材料都有同等价值。和任务目标直接相关的材料应优先进入上下文；只提供背景、风格或历史约束的材料可以摘要化；完全无关的材料应该剔除。比如今天要讲新概念，就需要查看近期教学记录，避免重复讲“渐进披露”或“注意力机制”，但不需要把每篇旧讲解全文都塞进来。

**第三层：顺序与结构管理。**

模型会受到上下文排列方式影响。把任务目标、硬性约束、输入材料、输出格式清楚分区，比把所有内容混成一团更可靠。这就是为什么 prompt 常写成“角色设定、输入文件、输出路径、必要约束、示例”。结构越清晰，模型越容易判断什么是指令，什么是素材。

**第四层：阶段管理。**

复杂任务可以拆成多个阶段：先总结更改，再提炼概念，再生成讲解，再发布 HTML。每个阶段只接收上一个阶段的稳定产物，而不是把所有原始材料反复传给每个 Agent。这种渐进披露能降低认知负担，也能让错误更容易定位。

这里基于当日材料可确认的是：上下文管理同时出现在 Agent 协作和 LLM 架构教学中。前者关注怎样给 Agent 提供聚焦输入，后者关注模型怎样在上下文窗口中通过注意力使用信息。一个偏工程，一个偏原理，但共同点都是“信息不是越多越好，而是要可被正确使用”。

### 有趣的相关事例

有时让模型修改一段代码，直接贴整个项目并不如只提供相关文件、错误日志和目标行为有效。整项目看起来更完整，但模型可能被无关模块吸引，提出大范围重构。聚焦上下文反而像给医生提供清晰病历：症状、检查结果、用药史一目了然。

### 常见误区

**误区一：上下文越长，结果越好。**

长上下文提供了更多可能证据，也提供了更多噪声。质量关键在相关性和优先级，不是长度本身。

**误区二：只要写了约束，模型一定会遵守。**

约束如果被埋在大量材料中，或者和后文示例冲突，遵守率会下降。重要约束应该清晰、靠前、结构化，并在输出格式中得到呼应。

**误区三：上下文管理只是 prompt 技巧。**

它还涉及文件组织、阶段产物、日志摘要、检索策略、多 Agent 分工和失败恢复。它是工作流设计，不只是文字润色。

### 与前一天工作内容的关系

前一天讲过 Prompt 约束工程。上下文管理是 Prompt 约束工程的上游基础：如果输入材料混乱，再漂亮的输出格式约束也很难救回来。今天的材料还把上下文管理和多 Agent 串行审阅联系起来：每个审阅 Agent 只看与自己职责相关的材料，反馈质量会更稳定。

### 一个具体例子

任务：让 Agent 根据当天材料生成知识讲解。

**差的上下文管理：**

把所有仓库的全部文件、所有日报、所有旧 prompt、所有历史聊天一次性塞给 Agent，然后说“写今天的知识讲解”。

可能问题：

- Agent 不知道哪些文件才是今天输入；
- 可能重复讲昨天已经讲过的概念；
- 可能引用过期信息；
- 输出格式容易偏离当前自动化要求。

**好的上下文管理：**

提供：

- 今天的 `concept_relevance.md`；
- 当月 `knowledge_log`，用于避免重复；
- 任务 prompt，说明输出路径、字段、约束；
- 必要时提供昨天一篇讲解作为风格参考。

这样 Agent 的注意力会集中在“今天概念”“历史避重”“指定格式”三个关键点上。

### 与其他概念的关联

| 关联概念 | 关系说明 |
| --- | --- |
| 上下文窗口 | 上下文管理首先受模型可处理长度限制影响。 |
| 注意力机制 | 注意力决定模型如何在上下文中分配关注，上下文管理决定材料如何进入这个范围。 |
| 渐进披露 | 把复杂信息分阶段给出，是上下文管理的重要策略。 |
| 多 Agent 串行审阅 | 每个 Agent 使用聚焦上下文，可以分别检查不同风险。 |
| Prompt Engineering 中的上下文裁剪 | 裁剪是上下文管理在 prompt 层面的具体操作。 |

### 可以继续深入学习的问题

1. 怎样判断一段材料应该原文进入上下文，还是应该摘要后进入上下文？
2. 多 Agent 审阅中，每个 Agent 应该共享完整历史，还是只共享最终结论？
3. 检索增强生成（RAG）怎样帮助系统动态选择相关上下文？

## 三、可测试需求

### 概念名称

**可测试需求**（Testable Requirements）

### 概念属于哪个领域

**软件工程 / 需求分析 / 测试工程 / 系统设计**

它常出现在产品需求文档、技术设计文档、验收标准、后端接口设计、权限系统和自动化流程中。

### 难度评级

★★★★（四星 / 较难）

写一句“支持删除项目”很容易，写清楚“删除前如何确认、删除后哪些数据保留、哪些数据清理、重复删除会怎样、失败能否恢复、不同权限用户能否操作”就难得多。可测试需求要求我们把模糊愿望变成可验证条件。

### 概念讲解

同学，我们先看两句话：

```text
系统应该安全地删除 Project。
```

```text
当用户删除 Project 时，系统先进入软删除状态；只有项目所有者二次确认后，才清理该 Project 私有的 PDF、解析文本、向量索引、图谱文件、运行历史和临时产物；不得删除全局 papers 记录，也不得影响其他 Project 或其他用户的数据；重复触发删除应保持结果一致。
```

第一句话听起来正确，但几乎没法测试。什么叫安全？删哪些？谁能删？失败怎么办？第二句话虽然更长，却能拆成测试用例。

**可测试需求**的核心思想就是：需求不仅要表达目标，还要表达可观察、可判断、可复现的验收条件。它要让开发、测试、产品和 AI 编码 Agent 都能回答同一个问题：**做到什么程度才算完成？**

一个需求要变得可测试，通常需要补齐五类信息。

**第一类：触发条件。**

什么时候发生？谁触发？前置状态是什么？例如“项目所有者点击删除并完成二次确认后”比“删除项目时”更清楚。

**第二类：预期结果。**

系统应该产生哪些变化？例如 Project 状态变为软删除、私人资产进入清理队列、界面不再默认展示该 Project。

**第三类：边界条件。**

异常或特殊情况怎样处理？比如 Project 已经被删除时再次删除、清理过程中网络失败、部分资产不存在、用户没有权限。

**第四类：不变量。**

哪些东西绝对不能被破坏？今天材料中的“全局实体与私有派生资产分离”就是典型不变量：`papers` 保存可复用论文身份，Project 私有资产可以清理，但全局论文身份不能被误删。

**第五类：验收方式。**

怎样验证？可以是单元测试、集成测试、权限测试、幂等测试、日志检查、UI 检查或人工验收清单。

可测试需求的价值在于，它会提前逼我们面对模糊地带。很多 bug 不是程序员不会写代码，而是需求没有说明“重复点击会怎样”“部分失败算成功还是失败”“删除共享数据是否允许”。当这些问题在需求阶段被写清楚，后面的实现和测试都会稳很多。

今天材料中提到的发布前置校验、失败分类与重试策略、软删除与二次确认、幂等操作，都属于可测试需求的好朋友。它们共同把一句宽泛的“系统要可靠”，拆成可执行、可观察、可验证的工程条款。

### 有趣的相关事例

“按钮要快”不是可测试需求，因为没人知道多快算快。“点击搜索后，95% 请求在 800ms 内返回首屏结果；超时 3s 显示重试入口并记录错误码”才可测试。可测试需求就像给抽象形容词装上尺子。

### 常见误区

**误区一：需求写得越短越高级。**

简洁很好，但不能牺牲可验证性。关键边界不写，后面一定会在开发、测试或线上事故中补课。

**误区二：测试是测试工程师后面再想的事。**

如果需求本身不可测试，测试工程师只能猜。可测试性应该在需求阶段就设计进去。

**误区三：只测试成功路径就够了。**

复杂系统的风险常在失败路径：权限不足、网络中断、重复提交、部分清理失败、历史数据不一致。可测试需求必须覆盖这些边界。

### 与前一天工作内容的关系

前一天讲过发布前置校验。发布前置校验其实就是可测试需求的一种实践：不要只写“发布成功”，而要拆成“网络可达、SSH 认证可用、远端仓库可读写、manifest 指向真实文件、页面可访问”等可验证条件。

### 一个具体例子

需求：Project 私人资产清理。

可测试版本可以写成：

| 测试点 | 可验证要求 |
| --- | --- |
| 权限 | 只有 Project 所有者或授权管理员可以发起清理。 |
| 资产范围 | 清理 PDF、解析文本、向量索引、图谱文件、运行历史和临时文件。 |
| 保护范围 | 不删除全局 `papers` 身份记录，不影响其他 Project 的派生资产。 |
| 二次确认 | 用户确认影响范围后才执行不可逆清理。 |
| 软删除 | 默认先把 Project 标记为软删除，支持恢复窗口或审计。 |
| 幂等 | 同一清理请求重复执行，不应重复报错或误删额外数据。 |
| 部分失败 | 某个资产删除失败时记录错误，并允许安全重试。 |

这张表直接就能变成开发检查清单和测试用例。

### 与其他概念的关联

| 关联概念 | 关系说明 |
| --- | --- |
| 失败分类与重试策略 | 可测试需求要说明不同失败类型如何处理。 |
| 幂等操作 | 重复执行是否安全，是后台流程需求的重要验收点。 |
| 软删除与二次确认 | 把高风险操作拆成可恢复、可确认的阶段。 |
| 全局实体与私有派生资产分离 | 为清理需求提供数据边界和不变量。 |
| 发布前置校验 | 把发布成功拆成多个可测试前置条件。 |
| 多 Agent 串行审阅 | 不同 Agent 可以分别审查权限、边界、幂等和失败场景。 |

## 概念之间的整体关联总结

今天三个概念可以串成一条很自然的链路。

**Transformer 前向推理流程**告诉我们：LLM 的生成不是凭空出现，而是把输入文本转成 token、向量和概率分布的计算过程。这个视角让我们明白，输入材料会真实影响模型后续输出。

**上下文管理**接着回答：既然输入会影响输出，那自动化 Agent 就不能随意堆材料，而要选择、排序、裁剪和分阶段组织上下文，让模型看到最相关、最可靠、最有优先级的信息。

**可测试需求**再进一步回答：当 Agent 或人类要实现复杂系统时，光有上下文还不够，任务目标也必须写成可验证条件。只有验收标准、边界条件、失败分类和不变量都清楚，系统才能从“看起来会做”变成“可以确认做对”。

所以今天的主线是：

```text
模型如何计算 -> Agent 如何接收上下文 -> 工程目标如何被验证
```

这也是许多 AI 工程实践的底层逻辑：理解模型，不是为了崇拜模型，而是为了更好地设计输入、流程和验收标准。

## 适合放入学习日报的精简结论

今天的三个知识点共同说明：大语言模型的输出来自一条可拆解的前向推理流水线；Agent 自动化要通过上下文管理控制输入质量；软件工程中的需求则应写成可测试、可验收、可覆盖边界的条件。把这三者连起来看，可靠的 AI 工作流不是“让模型多想想”，而是让输入、过程和验收都变得清晰可控。
