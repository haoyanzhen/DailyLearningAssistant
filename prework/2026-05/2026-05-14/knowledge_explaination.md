## 一、Manifest 驱动的发布契约

### 概念名称

**Manifest 驱动的发布契约**（Manifest-driven Publishing Contract）

### 概念属于哪个领域

**软件工程 / 静态站点发布 / 前端工程**

它常见于静态网站、前端构建系统、插件系统、移动应用资源包、数据目录和自动化内容发布流程中。

### 难度评级

★★★（三星 / 中等）

它的核心思想不难：用一个清单文件告诉系统“有哪些资源、在哪里、怎么展示”。难点在于，清单文件一旦成为契约，就必须和真实资源保持一致，否则页面虽然存在，系统却可能找不到它。

### 概念讲解

同学，我们先想象一个图书馆。

一本书真的放在书架上，并不等于读者一定能找到它。读者通常要靠目录系统：输入书名，目录告诉你馆藏位置、分类号、是否可借。如果书已经上架，但目录没有登记，那它在物理上存在，在读者体验里却像“消失”了一样。

静态网站里的 `manifest.json` 就很像这个目录。

在今天的材料中，学习日报最终会生成类似 `daily_report/2026-05/2026-05-14-learning-report.html` 的静态页面。但光有 HTML 文件还不够。主页需要知道：

- 今天有哪些日报可以展示；
- 最新日报是哪一篇；
- 归档列表应该出现哪些入口；
- 点击导航时应该打开哪个路径；
- 某个日期的页面是否已经发布。

这些信息如果全靠人手工写进首页，容易漏、容易错，也不利于自动化。于是就需要一个结构化索引，比如 `daily_report/manifest.json`。它把“页面资源”变成“可被程序读取的发布目录”。

所谓**Manifest 驱动的发布契约**，就是：系统约定一个 manifest 文件作为资源索引，发布流程不仅要生成资源本身，还要同步更新 manifest；访问方则按照 manifest 来寻找、展示和导航这些资源。

这里有两个关键词。

第一个关键词是**驱动**。它表示 manifest 不是附属说明文档，而是程序真正依赖的数据源。主页可以读取 manifest 来生成日报列表，默认打开最新日报，或者构建月份归档。

第二个关键词是**契约**。它表示 manifest 的字段、路径、排序和语义必须稳定。比如如果约定每条日报记录包含 `date`、`title`、`path`，那发布流程就不能今天写 `path`，明天改成 `url`，除非所有读取方一起升级。契约的价值正在于：生成方和消费方不用每次重新猜对方是什么意思。

这个概念背后的逻辑可以分成三层。

**第一层：资源存在不等于资源可达。**

HTML 文件放在目录里，只说明它被生成了；manifest 记录了它，才说明站点导航可以发现它。自动化发布要关心“文件生成”和“入口可达”两个条件。

**第二层：索引文件把离散资源组织成系统。**

一堆日报 HTML 是许多孤立页面；manifest 把它们按日期、标题、路径组织起来，主页才能形成归档、默认展示和导航。

**第三层：契约让自动化阶段可以衔接。**

日报生成 Agent 负责写 HTML，发布 Agent 负责更新 manifest，主页负责读取 manifest。只要契约稳定，不同阶段就可以独立工作，又能在最终网站里合起来。

### 有趣的相关事例

很多软件都有 manifest。浏览器扩展有 `manifest.json`，告诉浏览器插件名称、权限、脚本入口和图标。前端构建工具可能生成资源 manifest，把带哈希的文件名映射到原始资源。移动应用也有 manifest，用来描述应用权限和入口 Activity。

这些场景共同说明一件事：当资源越来越多，系统不能只靠“文件在那里”来工作，还需要一个可靠目录告诉程序“应该怎样理解这些文件”。

### 常见误区

**误区一：manifest 只是给人看的列表。**

不是。真正的 manifest 通常是给程序读的。它的字段命名、路径格式和排序规则会直接影响网站行为。

**误区二：页面生成成功就等于发布成功。**

如果页面存在但 manifest 没更新，主页可能没有入口；如果 manifest 指向不存在的页面，用户会遇到 404。发布成功必须同时满足“资源存在”和“索引正确”。

**误区三：manifest 可以随便改结构。**

一旦主页、脚本或其他自动化任务依赖 manifest，它就变成接口契约。修改结构要像修改 API 一样谨慎。

### 与前一天工作内容的关系

前一天讲过“契约式开发”。今天的 manifest 正是契约式开发在静态发布中的落点：HTML 页面是资源，manifest 是资源入口契约，主页是契约消费者。三者一致，发布闭环才完整。

### 一个具体例子

假设今天生成了日报：

```text
daily_report/2026-05/2026-05-14-learning-report.html
```

如果 manifest 里没有这条记录，首页的“最新日报”可能仍然停留在 2026-05-13。用户打开网站时，看不到今天的内容。

如果 manifest 写了这条记录，但路径拼成：

```text
daily_report/2026-05/2026-05-14-report.html
```

主页会显示入口，但点击后找不到文件。

所以可靠发布必须同时检查：

- HTML 文件确实存在；
- manifest 有对应日期记录；
- manifest 中的路径指向真实文件；
- 首页读取 manifest 后能展示正确入口。

这就是 manifest 作为发布契约的实际意义。

### 与其他概念的关联

| 关联概念 | 关系说明 |
| --- | --- |
| 静态资源索引 | manifest 是静态资源索引的一种具体形式。 |
| 状态可追溯性 | manifest 记录发布入口，可以帮助追踪某一天内容是否进入站点导航。 |
| 中间产物留痕 | `prework` 解释内容来源，manifest 解释内容入口，两者共同支撑可复盘发布。 |
| 多阶段自动化流水线 | 每个阶段通过稳定文件契约交接，manifest 是发布阶段的关键交接物。 |
| 契约式开发 | manifest 的字段和语义就是生成方与消费方之间的契约。 |

1. 一个日报站点的 manifest 至少应该包含哪些字段，才能支持归档、默认展示和搜索？

3. 如果以后日报数量达到几千篇，manifest 是否还适合一次性加载？

## 二、发布前置校验

### 概念名称

**发布前置校验**（Pre-release Validation）

### 概念属于哪个领域

**软件工程 / DevOps / 自动化发布**

它常用于 CI/CD、静态站点部署、包发布、数据库迁移和任何“操作一旦执行就会影响外部可见状态”的流程中。

### 难度评级

★★★★（四星 / 较难）

它不只是多跑几个检查命令，而是要理解发布链路里哪些条件是“必须先确认”的：网络是否可用、认证是否有效、远程仓库是否可写、本地改动是否完整、生成结果是否能被访问。

### 概念讲解

同学，我们先用寄快递来理解发布。

你把礼物包装好了，这相当于本地文件已经生成。可是要让朋友真的收到礼物，还需要确认：地址是否正确、快递站是否营业、运费是否支付、包裹是否能被揽收。否则你只能说“我把礼物包好了”，不能说“我寄出去了”。

软件发布也是这样。

在今天的材料中，发布 prompt 增加了一个要求：执行 `git push` 前，要检测网络与 GitHub SSH 认证。这就是典型的发布前置校验。它要解决的问题是：自动化任务不能只在本地生成文件，然后假装远程同步已经完成。

对于依托 GitHub Pages 的静态网站来说，完整发布链路大致是：

1. 生成日报 HTML；
2. 更新 manifest 和首页导航；
3. 提交到 Git；
4. 推送到 GitHub；
5. GitHub Pages 根据远程仓库内容提供访问。

前两步发生在本地，后几步涉及远程网络和认证。如果网络断开、SSH key 失效、远程仓库权限变化，`git push` 就会失败。此时本地看起来一切完成，但线上用户看不到新日报。这就是半完成状态。

**发布前置校验**的核心思想是：在执行关键发布动作前，先检查那些可能导致发布失败的外部条件，尽早发现问题，并把失败原因明确暴露出来。

它背后的逻辑有三层。

**第一层：把失败提前。**

如果 SSH 认证不可用，越早发现越好。不要等文件都生成、提交都写好、最后一步才发现推不上去。提前发现可以减少无效工作，也能让自动化报告更清楚。

**第二层：区分本地完成和远程完成。**

本地生成成功不是发布成功。发布成功必须考虑远程可见性。对于 GitHub Pages，远程仓库是否收到新提交是非常关键的一环。

**第三层：让自动化有诚实的终态。**

一个好的自动化任务应该明确说出自己处于哪种状态：已生成但未推送、已提交但推送失败、已推送等待 Pages 部署、还是完整发布。这样用户才能知道下一步要做什么。

### 有趣的相关事例

很多线上事故不是因为代码完全错误，而是因为发布前少了一个简单检查。比如配置文件里指向测试环境数据库，迁移脚本没有检测表是否已存在，部署机器没有权限拉取镜像。真正成熟的发布系统，会把这些容易被忽略的前提条件变成自动检查。

这也是为什么一些团队会说：“发布脚本要像飞行员起飞前检查清单一样工作。” 不是因为飞行员不聪明，而是因为复杂系统里，靠记忆很容易漏步骤。

### 常见误区

**误区一：前置校验会拖慢发布，所以能省就省。**

几秒钟的检查通常比一次失败发布便宜得多。尤其是自动化任务无人值守运行时，清楚失败原因比速度更重要。

**误区二：只要 `git push` 失败时再报错就够了。**

失败时报错当然需要，但前置校验可以给出更可读、更可操作的提示。例如“SSH 认证失败”比一大段 Git 错误更容易处理。

**误区三：校验越多越好。**

校验要针对关键风险。过多低价值检查会让流程复杂、噪声变大。好的前置校验应该检查那些失败概率不低、失败后影响明确、并且能提前判断的条件。

### 与前一天工作内容的关系

前一天讲过“状态机与并发锁”，其中一个重点是避免系统进入混乱状态。发布前置校验也有类似目的：它通过提前检查外部条件，减少“本地已完成但线上未更新”的模糊状态。

### 一个具体例子

假设自动化任务准备发布今天的学习日报。它可以在推送前检查：

```bash
ssh -T git@github.com
git remote -v
git status --short
```

这些检查分别回答：

- 当前机器能否通过 SSH 连接 GitHub；
- 当前仓库推送目标是不是预期远程；
- 本地是否存在需要提交的改动。

如果 SSH 认证失败，任务就不应该轻描淡写地说“日报已发布”。更准确的状态应该是：

> 日报文件已生成，本地改动已准备好，但 GitHub SSH 认证失败，远程发布未完成。

这句话虽然不如“全部成功”好听，但它对用户更有用。

### 与其他概念的关联

| 关联概念 | 关系说明 |
| --- | --- |
| 自动化任务的操作安全边界 | 前置校验定义哪些条件不满足时不应继续执行高风险操作。 |
| GitHub Pages 发布 | 远程推送成功是静态页面线上可见的重要前提。 |
| 状态可追溯性 | 校验结果可以写入日志，让发布失败原因可追踪。 |
| CI/CD | 现代持续集成和持续部署流程大量依赖发布前检查。 |
| 半完成状态 | 前置校验的目标之一就是减少或清楚标记半完成状态。 |

### 可以继续深入学习的问题

1. 发布脚本应该如何区分“警告”和“阻断错误”？
2. 哪些前置校验适合在本地执行，哪些更适合交给 CI 执行？
3. 如果网络偶发失败，自动化任务应该立即失败、重试，还是等待人工确认？

## 三、Prompt 约束工程

### 概念名称

**Prompt 约束工程**（Prompt Constraint Engineering）

### 概念属于哪个领域

**大语言模型 / Agent 工作流工程 / 软件工程**

它关注的不是单次问答写得漂不漂亮，而是如何把稳定规则、边界条件和操作流程写进 Agent 的提示词，让自动化行为更可控。

### 难度评级

★★★★（四星 / 较难）

普通提示词强调“让模型完成任务”，Prompt 约束工程强调“让模型在正确边界内完成任务”。它需要同时理解 LLM 的行为特点、工程流程、风险边界和验证方式。

### 概念讲解

同学，我们先区分两个很像但不完全一样的概念：提示词工程和 Prompt 约束工程。

普通提示词工程像是在说：

> 请帮我写一篇学习日报，语气亲切，结构清晰。

Prompt 约束工程更像是在说：

> 你要先读取哪些文件，只能修改哪些文件；如果输入文件不存在，要报告阻塞；生成日报后要更新 manifest；推送前必须检测网络和 SSH 认证；不允许覆盖概念提炼文件；如果远程推送失败，要明确说明本地和远程状态。

也就是说，它不是只追求输出质量，而是把工程规则写成模型必须遵守的行为边界。

为什么这件事重要？因为 LLM 很擅长根据上下文补全下一步，但它不天然知道你的项目规则。你说“发布日报”，模型可能会理解成“生成一个 HTML 文件就够了”；你说“可靠发布日报”，它仍然可能不知道必须更新 manifest、检查导航、提交 Git、验证推送。只有把这些规则明确写进 prompt，Agent 才更可能按你的工程流程行动。

今天材料里提到发布 prompt 增加了 `git push` 前的网络与 GitHub SSH 认证检测要求。这就是 Prompt 约束工程的典型例子：把一次人工经验沉淀成以后每次自动化任务都要执行的规则。

Prompt 约束工程背后的逻辑可以分成四层。

**第一层：把隐性经验显性化。**

人类开发者可能知道“推送前要确认 SSH 认证”，但 Agent 不一定知道。写进 prompt 后，这条经验就从脑子里的习惯变成可复用的规则。

**第二层：把目标拆成可检查步骤。**

“发布成功”太抽象。Prompt 应该把它拆成生成文件、更新索引、检查 Git 状态、验证网络认证、推送远程、报告结果等步骤。

**第三层：给失败路径命名。**

好的约束不只规定成功时做什么，也规定失败时如何诚实报告。比如输入缺失、认证失败、远程不可达、文件冲突，都应该有清楚处理方式。

**第四层：限制 Agent 的操作边界。**

自动化 Agent 可能拥有文件修改、命令执行、网络访问等能力。Prompt 约束要告诉它哪些文件可以改、哪些文件不能覆盖、哪些操作需要确认、哪些状态不能假装成功。

### 有趣的相关事例

可以把 Prompt 约束工程想象成给一位很聪明的实习生写工作手册。你不需要告诉他“每个字怎么打”，但要告诉他工作边界：从哪里拿材料，结果放在哪里，遇到缺文件怎么办，哪些动作不能擅自做，交付时必须说明什么。

LLM Agent 也是类似的。它能力强，但上下文和规则需要被清楚提供。越是重复运行的自动化，越应该把关键经验沉淀到 prompt 里。

### 常见误区

**误区一：Prompt 越长越可靠。**

不一定。长 prompt 如果结构混乱、规则互相矛盾，反而会降低可靠性。好的约束应该清晰、分层、可执行。

**误区二：Prompt 写了规则，Agent 就一定不会犯错。**

Prompt 约束能显著降低风险，但不能替代测试、日志、权限控制和人工审核。对于高风险操作，还需要工具层和流程层保护。

**误区三：只约束输出格式，不约束操作过程。**

很多自动化失败不是因为最终文本格式错，而是因为读取了错误输入、覆盖了不该覆盖的文件、跳过了验证步骤。过程约束同样重要。

### 与前一天工作内容的关系

前一天讲过“契约式开发”。Prompt 约束工程可以理解为“给 Agent 的契约式开发”：人类把输入、输出、边界、失败处理和验收标准写清楚，Agent 在这个契约内执行任务。

### 一个具体例子

假设原始 prompt 只有一句：

> 生成今天的学习日报并发布。

这太模糊。一个更像 Prompt 约束工程的版本会写：

```text
1. 读取 prework/YYYY-MM/YYYY-MM-DD/knowledge_explaination.md。
2. 生成 daily_report/YYYY-MM/YYYY-MM-DD-learning-report.html。
3. 更新 daily_report/manifest.json，并确保最新日报指向当天文件。
4. 不要覆盖 prework 中的输入文件。
5. git push 前先检测网络与 GitHub SSH 认证。
6. 如果 push 失败，必须说明本地文件是否已生成、提交是否已创建、远程是否未同步。
```

这个 prompt 的目标不是“显得更复杂”，而是把发布流程中的关键边界变成 Agent 可执行的步骤。

## 概念之间的整体关联总结

今天的三个概念可以组成一条很清楚的发布可靠性链路。

**Manifest 驱动的发布契约**解决“内容如何被站点发现”的问题。它告诉我们，生成 HTML 只是第一步，页面必须进入稳定索引，才能被主页导航和归档系统使用。

**发布前置校验**解决“内容能否真正同步到远程”的问题。它提醒我们，本地成功不等于线上成功，网络、认证和远程写入能力都属于发布闭环的一部分。

**Prompt 约束工程**解决“Agent 为什么会稳定执行这些步骤”的问题。它把 manifest 更新、发布前检查、失败报告等工程经验写进自动化规则，让一次经验变成后续流程的默认行为。

用一句话串起来就是：**manifest 定义发布入口，前置校验保障发布链路，prompt 约束让 Agent 每次都按这套可靠流程工作。**

## 适合放入学习日报的精简结论

今天的知识重点是“自动化发布的可靠性”。一个学习日报系统不能只生成 HTML，还要用 manifest 维护可导航的资源索引；不能只在本地完成文件修改，还要在发布前校验网络和 GitHub SSH 认证，避免远程未同步的半完成状态；也不能只依赖 Agent 临场发挥，而要把这些工程边界写进 prompt，形成可重复、可检查、可追溯的自动化流程。
