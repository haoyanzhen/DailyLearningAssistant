# 2026-05-20 知识讲解

## 今日知识主线

今天我们从日常学习日报自动化链路中提炼出三个关键概念，它们共同回答了这个问题：**如何让大语言模型（LLM）在自动化流水线中既发挥创造力，又保持可靠？** 我们从最基础的约定开始，逐步深入到状态验证和生成边界控制，带你理解一条工程化的“AI 自动化”路径。

---

## 概念一：结构化输出契约（★★☆☆☆）

### 领域
计算机科学 / 软件工程 / 大语言模型工程

### 亲和完整讲解
想象你请一个朋友帮你写一段摘要，但朋友每次给你的格式都不一样：有时是列表、有时是散文、有时还画了图。你要把这些摘要自动合并到表格里，就很头疼。**结构化输出契约** 就是提前和 LLM 说好：“请用下面这个 JSON 格式返回结果，字段名、类型、甚至表格的行列合并都要按约定来。”

这里的“契约”有两层意思：
1. **格式契约**：比如我们要求 LLM 返回一个 JSON，包含 `title`、`date`、`summary` 等字段，并且 `summary` 里必须包含一个 HTML `<table>` 标签，且特定行要设置 `rowspan="3"`。
2. **校验契约**：LLM 返回后，我们的脚本会去解析这个 JSON，检查每个字段是否存在、格式是否正确。如果不符合，就拒绝使用这次输出，而不是直接写入文件。

### 一个具体例子
在 `DailyLearningAssistant` 项目的知识讲解任务中，脚本向外部 LLM 发送 prompt 时，明确要求返回以下结构：
```json
{
  "concept_name": "结构化输出契约",
  "difficulty": 2,
  "explanation": "...HTML 表格..."
}
```
如果 LLM 返回了没有 `explanation` 字段的 JSON，或者表格缺少 `rowspan`，脚本会抛出校验错误，不会让不符合约定的内容进入知识日志。

### 与其他概念的关联
- **LLM 输出校验**：结构化输出契约只是“约定”，而 LLM 输出校验是“落实”。脚本中的 JSON 解析和字段检查就是把契约变成强制门槛。
- **配置驱动自动化**：契约中要求的具体 schema 可以通过配置文件传入，使同一脚本能适配不同模型或任务。

### 可继续深入学习的问题
- 如何设计一个既足够严格又不至于让 LLM 频繁失败的 schema？
- 对于嵌套 JSON 和数组字段，如何高效校验？
- 当 LLM 输出包含富文本（如 HTML、Markdown）时，如何安全地解析和渲染？

## 概念二：状态校验（★★★☆☆）

### 领域
计算机科学 / 软件工程 / 自动化运维

### 亲和完整讲解
你有没有过这种经历：电脑提示“网络已断开”，但你明明插着网线；或者你点击“推送”，系统却告诉你“本地落后于远端，请先拉取”。**状态校验** 就是让自动化系统在做关键动作之前，先“自检”一遍，确认自己处在正确的状态下再行动。

状态校验的核心是**将隐式假设变成显式检查**。比如在自动 git push 之前，我们通常假设：
- 网络连通
- SSH 密钥有效且远端可读
- 本地分支与远端分支的关系是“领先”或“一致”，而不是“落后”
- 当前分支正确（比如是 `main`）

如果没有这些检查，一旦某个假设不成立，程序就会报错甚至把错误的数据推上去。

### 一个具体例子
`DailyLearningAssistant` 的自动推送脚本在执行 `git push` 前，会依次检查：
1. 能否解析 GitHub 的 DNS？
2. SSH 连接是否能成功认证？
3. 远端仓库是否存在且可写入？
4. 本地 `main` 分支是否 ahead 于远端 `main`？
5. 如果本地落后于远端，则停止执行并输出警告。

只有全部通过，才执行 `git push`。这个行为就是“防护式自动化”的具体体现。

### 与其他概念的关联
- **防护式自动化**：状态校验为防护式自动化提供了决策依据——只有状态安全时才继续。
- **发布前置同步**：状态校验中的 ahead/behind 检查直接决定了是否需要先同步再推送。

### 可继续深入学习的问题
- 对于长时间运行的自动化任务，状态可能发生变化，如何实现“重校验”？
- 哪些状态应该缓存，哪些应该实时检查？
- 如何设计一个可扩展的状态检查框架，让不同模块插拔自己的检查项？

## 概念三：防幻觉生成边界（★★★★☆）

### 领域
大语言模型 / Agent 工作流工程 / 软件工程

### 亲和完整讲解
LLM 天生会“编故事”——当它不知道答案时，它会根据上下文猜测，而且说得像真的一样。这在闲聊时很有用，但在自动化流水线中可能是灾难。**防幻觉生成边界** 就是主动给 LLM 画一个圈，告诉它：“只许用圈内的信息来源，圈外的不许编。”

这个“圈”由两部分组成：
1. **输入证据边界**：只有我们明确提供给 LLM 的摘要、事实、数据，才允许出现在输出中。
2. **输出约束边界**：如果证据不足，LLM 不应该补写，而应该输出“无相关信息”或直接失败。

在 `DailyLearningAssistant` 中，这个原则被严格执行：
- 知识讲解任务：如果 LLM 调用失败，Codex（主控脚本）不能自己补写一篇内容，必须记录失败状态。
- 邮件推送任务：prompt 中明确要求“不要猜测收件人的现实状态（如是否已读、是否忙碌），只基于日报摘要和今天的概念生成邮件”。

### 有趣例子
假设今天没有实时天气数据，但 LLM 在写邮件时自己加了一句“今天气温24°C，适合户外运动”。这看起来无害，但它打破了来源可追溯性。如果用户第二天发现气温不对，他们不会质疑 LLM，而会怀疑整个系统的数据质量。防幻觉生成边界要求：如果天气数据未提供，就写“今天天气信息暂缺”，而不是编一个。

### 常见误区
有人觉得“LLM 编一点也没关系，反正用户看不出来”。但一旦出现事实错误，整个自动化系统的信任度就会崩塌。防幻觉不是“找理由限制 LLM”，而是**保护自动化流水线的可审计性**。

### 与其他概念的关联
- **数据血缘**：有了数据血缘，系统才知道哪些信息是可信的输入来源，从而画出“圈”的边界。
- **Prompt 风格档案**：邮件可以有随机风格，但风格不能突破事实边界——不能因为要“亲切”就编造信息。

### 可继续深入学习的问题
- 如何通过 RAG（检索增强生成）让 LLM 只引用指定文档？
- 如何设计“证据绑定提示词”让 LLM 强制引用来源？
- 自动化系统中，当输入缺失时，怎样优雅地降级（而非编造）？

## 整体关联总结

今天的三个概念从**约定（结构化输出契约）** 到**检查（状态校验）** 再到**限制（防幻觉生成边界）**，构成了一条“可靠 AI 自动化流水线”的底层逻辑：
- 首先和 AI 约定好沟通格式，让后续处理不再混乱；
- 然后在系统层面检查环境状态，确保动作执行在安全条件下；
- 最后在内容生产层面约束 AI 的想象力，让成果只来源于真实证据。

这三者缺一不可：没有契约，输出不可解析；没有校验，动作可能出错；没有边界，内容可能失真。

## 适合放入学习日报的精简结论

> **今天学到了三个关键理念：**
> 1. 结构化输出契约：让 LLM 的输出像 API 一样可解析、可校验。
> 2. 状态校验：自动化动作前先自检，避免在错误状态下执行。
> 3. 防幻觉生成边界：限制 LLM 只使用真实输入，不编造信息。
> 它们共同把 LLM 从“创意生成器”变成“可信自动化工具”。
