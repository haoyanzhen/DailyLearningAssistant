# 自动化任务 3：每日知识讲解生成

## 1. Agent 角色设定

你是“知识讲解生成调度员”。你的职责是调用本项目 `config.json` 中 `llm` 配置的外部大模型来完成正文生成。Codex 只负责检查输入、执行脚本、确认输出文件已正确落盘；知识讲解正文和当月教学记录内容应由配置的 LLM 生成。

## 2. 工作内容详细指定

目标日期使用当前运行日期，格式为 `YYYY-MM-DD`。

读取：

```text
./prework/YYYY-MM/YYYY-MM-DD/concept_relevance.md
./knowledge_log/YYYY-MM-knowledge-log.md
```

开始生成前，必须先验证上游任务 2 的输出已经完成：

```text
./prework/YYYY-MM/YYYY-MM-DD/concept_relevance.md
```

如果 `concept_relevance.md` 缺失、为空或不可读取，应停止本任务并明确记录原因；不要调用外部 LLM，不要生成或覆盖 `knowledge_explaination.md`，也不要更新当月教学记录。

根据其中的概念清单和概念关联，以及过往知识点讲解历史，调用脚本：

```bash
python3 scripts/generate_knowledge_explaination.py
```

并通过 OpenAI Chat Completions 兼容接口生成详细知识讲解，保存为：

```text
./prework/YYYY-MM/YYYY-MM-DD/knowledge_explaination.md
```

同时生成或更新当月教学记录：

```text
./knowledge_log/YYYY-MM-knowledge-log.md
```

外部 LLM 生成内容时，每个概念的讲解应包含：

- 概念名称
- 概念属于哪个领域
- 难度评级（用五颗星评分制来表示）
- 用亲和、像是教导学生一样的语气完整地解释概念背后的理念、原理和逻辑
- 有趣的相关事例（可选）
- 常见误区（可选）
- 与前一天工作内容的关系（可选）
- 一个具体例子
- 与其他概念的关联
- 可以继续深入学习的问题，或开放性问题

整篇文件还应包含：

- 日期
- 今日知识主线
- 概念之间的整体关联总结
- 适合放入学习日报的精简结论

月度教学记录要求：

- 每月一张表，文件名固定为 `YYYY-MM-knowledge-log.md`。
- 表格使用 HTML `<table>`，因为日期列需要对每日三条概念记录使用 `rowspan="3"` 跨行合并。
- 表格字段固定为：`日期`、`概念`、`领域`、`难度`、`一句话解释`、`下一次讲解参考`。
- 日期文字必须链接到当天日报，例如：

  ```html
  <td rowspan="3">
    <a href="../daily_report/YYYY-MM/YYYY-MM-DD-learning-report.html">YYYY-MM-DD</a>
  </td>
  ```

- 每天固定写入 3 行，对应当天三个正式讲解概念。
- 如果同一天记录已存在，应更新该日期块，不要重复追加。
- 当月表格按日期倒序排列，最新日期在最上方。
- `一句话解释` 应控制为简明摘要，适合主页表格阅读。
- `下一次讲解参考` 用于提示后续日的知识更新方向，可以写前置概念、关联概念、工程实践问题或理论延伸。

## 3. 必要约束

- 必须优先使用 `scripts/generate_knowledge_explaination.py` 调用 `config.json` 中配置的 LLM，不要默认由 Codex 直接撰写 `knowledge_explaination.md` 正文。
- 如果 `config.json` 不存在、`llm` 配置缺失、API key 仍为占位符、网络不可用或 LLM 调用失败，应停止本任务并明确记录失败原因，不要回退为 Codex 自行生成正式讲解。
- 脚本默认使用当前运行日期；补跑历史日期时可使用：

  ```bash
  python3 scripts/generate_knowledge_explaination.py --date YYYY-MM-DD
  ```

- LLM 接口需兼容 OpenAI Chat Completions 响应格式，即返回 `choices[0].message.content`。
- 解释不可出现谬误，不要为了生动而牺牲技术正确性。
- 遇到不确定的概念背景，应明确写“这里基于当日材料可确认的是……”。
- 输出应使用 Markdown。
- 文件名按当前项目既定拼写保存为 `knowledge_explaination.md`，不要改成其他拼写。
- 不要覆盖 `concept_relevance.md`。
- 不要把完整长篇讲解塞入 `knowledge_log`，教学记录只保存可检索、可展示的表格摘要。
- 只有在当日 `concept_relevance.md` 已存在、非空且可读取后，才允许生成或更新 `knowledge_explaination.md` 和 `knowledge_log/YYYY-MM-knowledge-log.md`。

## 4. 示例

示例输出路径：

```text
./prework/2026-05-11/knowledge_explaination.md
```

示例内容：

```markdown
# 2026-05-01 知识点讲解

## 提示词工程

### 概念名称
**提示词工程**（英文：Prompt Engineering）

### 概念属于哪个领域
**人工智能（特别是自然语言处理）** + **人机交互**。它既是一门技术，也是一种沟通艺术。

### 难度评级
⭐⭐⭐（三星 / 中等）
*解释：入门很容易，你只要会打字就能开始。但要想稳定、高效、创造性地引导顶尖模型完成复杂任务，需要大量练习和对模型特性的理解，所以给三颗星。*

### 概念讲解

好的，同学，咱们先别被“工程”两个字吓到。你可以把**提示词工程**想象成：**学会向一个聪明、诚实、但有点“死脑筋”的超级助手准确、高效地提问的艺术**。

**理念**：这个超级助手（也就是大型语言模型，比如我DeepSeek）肚子里有海量的知识，但它不会读心术。你给它一个“提示词”（就是你的提问或指令），它就会尽力去“接话”，完成这个提示词所引导的任务。**你的提示词质量，直接决定了它回答的质量。** 提示词工程的核心理念就是：**不要指望模型来猜你想要什么，而要用最清晰、最结构化的方式把你想要的告诉它。**

**原理**：你可能好奇，它为什么这么“听提示词的话”？从技术原理上说，语言模型在训练中学到的是一种“文字接龙”的概率分布。你给它一段话（提示词），它会根据这段话，一个字一个字地预测接下来最可能出现的字。所以：
- 如果你给的提示词模糊（“写首诗”），模型发现“诗”后面可以接很多种开头，于是可能随机选一个“平平无奇”的开头。
- 如果你给的提示词非常具体（“写一首关于落日的五言绝句，表达思念之情，风格忧伤”），模型看到这么多约束，它预测出符合这些约束的文字的概率就大大升高了。

**逻辑**：提示词工程遵循一个简单的逻辑：**输入决定输出**。就像你在厨房里：你放盐（输入），菜就咸；你放糖，菜就甜。你想让模型输出什么类型的答案，就要在提示词里给它对应的“配方”：角色、格式、示例、步骤、约束条件……

**所以，学习提示词工程，本质上就是学习写“好配方”**。常见的配方技巧包括：
1.  **明确角色**：“你是一位资深律师” → 模型会用法言法语。
2.  **给出示例**：先给一两个例子示范你想要的“输入-输出”模式。
3.  **分解步骤**：“先分析原因，再提出对策” → 让模型按你的逻辑思考。
4.  **设定格式**：“输出JSON”或“用表格列出”。
5.  **加入约束**：“如果不知道就说不知道，不要编造”。

**要记住哦，你不是在编程，而是在和AI“沟通”。** 你的武器就是日常语言，只是需要把它用得更精准、更有结构。

## 有趣的相关事例

**事例1：画圆圈的悖论**
有人问一个早期的模型：“请画一个圆圈。‘画’字要用中文拼音首字母‘h’代替。” 模型回答：“请h一个圆圈。” —— 模型只是机械地把“画”字替换成了它的拼音首字母，而没有理解整个句子的意图。早期的提示词需要非常小心歧义，尽管现在的模型理解能力已经强多了，但这个故事还是告诉我们“模型如何机械地处理指令”。

**事例2：超级猜谜家**
有人给模型一个提示词：“你现在是世界上最擅长猜字谜的人。接下来我会把电影名称拆解成几个特征，每说一个特征，你就猜一次。规则是：不许一次猜中，要故意先错两次，最后再猜对。开始：第一部电影，特征1：太空。” 模型按照这个“角色+规则+步骤”的提示词，完美地扮演了一个调皮又聪明的猜谜者。这就是提示词工程的魅力——你可以编程般控制模型的行为。

### 常见误区

1.  **误区一：越客气、越啰嗦越好。**
    -   *错例*：“嗨，你好，亲爱的助手，请问如果你有时间的话，可不可以非常友好地帮我……写一封感谢信？不胜感激！”
    -   *正解*：直接给出清晰指令即可。过度客气浪费tokens且可能引入干扰。直接说：“写一封给张老师的感谢信，感谢他指导我论文。”
2.  **误区二：提示词必须一次性完美写好。**
    -   *错例*：写了一个复杂的提示词，模型输出不理想就灰心丧气。
    -   *正解*：**迭代优化**是核心！先写个简单版本，看看输出，再逐步增加约束、格式、示例。完美提示词往往是“改”出来的。
3.  **误区三：忽略模型的限制。**
    -   *错例*：让模型计算“两个18位数相乘”，模型可能因为精度问题给个近似值。
    -   *正解*：需要知道模型擅长“语言理解”而非“精确数学计算”，对于复杂计算，可以让它写代码或使用计算器工具（如果支持）。
4.  **误区四：认为提示词工程就是“咒语”，固定一种写法。**
    -   *正解*：不同的模型（GPT-4 vs. Claude vs. DeepSeek）对提示词的敏感度不同。同一个提示词效果可能不同。需要根据具体模型调整。但清晰、具体、结构化是**通用原则**。

### 与前一天工作内容的关系

1.  **写周报**：
    -   **差提示词**：“帮我写周报。”
    -   **好提示词（提示词工程）**：“请根据以下散乱记录，帮我整理成一份结构化的项目周报。要求包含：『本周进展』『遇到的问题』『下周计划』三部分，用Markdown列表格式。记录如下：周一修复了登录bug，周二开会讨论新功能，周三收到客户关于加载慢的投诉……‘’
2.  **分析客户反馈**：
    -   **差提示词**：“分析这些邮件。”
    -   **好提示词**：“你是一名客户体验分析师。请逐条阅读以下10条客户反馈，然后：1. 统计出现最多的3个问题关键词；2. 摘出情绪最消极的两条反馈原文；3. 用一段话总结主要改进方向。反馈内容：……”
3.  **写Python脚本**：
    -   **差提示词**：“写个脚本重命名文件。”
    -   **好提示词**：“写一个Python脚本，遍历当前目录下所有扩展名为.txt的文件，将文件名中的空格替换为下划线，但要保留日期部分（例如 'report 2023-01-01.txt` 改成 `report_2023-01-01.txt`）。请给出完整代码并添加注释。”

你看，通过应用提示词工程（给角色、给格式、给约束、给示例），你把一个模糊的请求变成了可执行的、高质量的助手任务，直接提升了你的工作效率。

### 一个具体例子

**背景**：你想让AI帮你写一封得体的请假邮件，但你不希望它写得太随便或太生硬。

**未使用提示词工程**：
> 提示词：“写一封请假邮件”
>
> AI输出：
> ```
> 主题：请假
>
> 你好，
> 我因为有事想要请假一天。请批准。
> 谢谢。
> ```
> *结果：过于简单，缺乏诚意和必要信息。*

**使用了提示词工程**：
>
> 提示词：“请帮我写一封给直属领导李经理的请假邮件。我请假的理由是重感冒并发高烧（附上医院挂号单照片）。请假时间是明天（2023年10月25日）一天。语气要**尊敬、得体**，同时**感到抱歉**。邮件需要包含：
> 1. 明确的请假申请
> 2. 充分但简洁的理由
> 3. 说明请假期间手头紧急任务的交接方案（把整理季度报表的任务请同事小王帮忙）
> 4. 承诺返岗后尽快补上工作
> 5. 表达感谢”
>
> AI输出：
> ```
> 主题：10月25日病假申请（高烧）
>
> 尊敬的李经理：
>
> 很抱歉打扰您。由于突发重感冒并高烧（已附上医院挂号单），医生建议休息一天。因此我恳请您批准明天（10月25日）请假一天。
>
> 关于手头的工作，明天的季度报表整理我已请同事小王帮忙跟进。返岗后我会第一时间处理遗留事项。
>
> 感谢您的理解！
>
> 此致
> 敬礼
> [你的名字]
> ```

**对比**：第二个输出完全符合你的要求，甚至可以直接使用。这就是提示词工程带来的巨大差异。

### 与其他概念的关联

| 关联概念 | 关系说明 |
| :--- | :--- |
| **大型语言模型（LLM）** | 提示词工程的应用对象。没有LLM，就没有提示词工程。 |
| **微调（Fine-tuning）** | 它们是**互补**的两种“教模型”的方式。<br>- 提示词工程：不动模型，只改输入，快速、低成本。<br>- 微调：用大量特定数据训练模型，改变模型参数，效果持久但贵且慢。 |
| **思维链（Chain-of-Thought, CoT）** | 提示词工程中**最核心的技巧之一**。要求模型“逐步思考”，极大地提高复杂推理任务的准确率。 |
| **少样本学习（Few-shot Learning）** | 另一种重要技巧。在提示词中**提供示例**，让模型从例子中学习你想要的输入-输出映射。 |
| **基于检索的生成（RAG）** | 高级应用。提示词工程可以用来调用**外部知识库**（给模型一个提示，让它先搜索相关知识，再回答问题）。 |
| **人工智能对齐（AI Alignment）** | 提示词工程可以帮助（尽管有限）让模型的输出更符合人类的价值观和安全要求（如：“回答要友善，不要侮辱用户”）。 |

### 可以继续深入学习的问题，或开放性问题

1.  **实践性问题**：对于一个你从未用过的新语言模型（比如未来的某款模型），你会设计一个怎样的“提示词测试集”来快速了解它的强项、弱项（比如数学、逻辑、创意写作、多语言能力）以及对提示词的敏感度？
2.  **设计性问题**：假设你要开发一个面向中小学生的“编程辅导AI”，但模型本身有提供完整代码的倾向（这会阻碍学生思考）。你将如何设计提示词系统（系统提示+用户提示）来**让它尽量引导学生自己找到答案，而不是直接给出代码**？
3.  **挑战性问题（对抗性提示词）**：既然提示词可以引导模型，那么恶意用户是否可以用精心设计的提示词（提示词注入攻击）来让模型绕过安全限制、输出有害内容？防御这种攻击有哪些提示词层面的策略？
4.  **哲学性问题**：随着模型变得越来越强大（理解上下文能力越来越强，甚至能直接读取脑电波），提示词工程是否会消失？或者说，它会演变成一种什么新的形式？（比如，模型主动提问澄清模糊需求？）
5.  **前沿技术问题**：自动提示词优化（Automatic Prompt Optimization, APO）技术——用另一个模型（或一个算法）来**自动编写和测试提示词**。你认为这会在多大程度上取代人类的手工提示词工程？又会催生哪些新的职业或技能？
