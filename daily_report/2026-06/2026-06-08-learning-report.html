设计文档与系统行为的一致性

Design Doc and System Behavior Consistency

一致性本身才是目标，文档与代码之间必须有一个被认定为“权威方”。

核心要点

  • 设计文档是系统“应该”怎样的规范来源，不是被动的记录。
  • 当文档与实际行为不一致时，需明确权威方，优先修正文档以恢复规范。
  • 一致性的目标是通过清晰的规范降低系统复杂度，避免团队成员对行为的猜测。

设计文档与系统行为的一致性

完整讲解

想象你买了一台微波炉，说明书上写着“按开始键后转盘会旋转”，但实际使用时转盘却不转。这时候有两个选择：一是修改说明书，承认转盘本来就不转；二是去修微波炉，让转盘转起来。设计文档与系统行为的一致性，就是在这两种选择中做出正确的工程判断。在正规的软件团队中，设计文档是系统行为的“规范来源”——它定义了系统“应该”怎么表现。当代码的实际表现和文档描述不一致时，那个不一致点就是一个“契约缺口”。修正这个缺口的正确方式，通常是改文档而不是改代码，因为文档代表了团队的共识和设计意图。反过来，如果代码有 bug 导致表现不符合文档，那就应该改代码。关键在于：一致性本身才是目标，而文档与代码之间必须有一个被认定为“权威方”。

原理逻辑

  1. 文档驱动开发：文档先行，代码实现文档。
  2. 文档是团队共识的显式化编码。
  3. 出现偏差时，修复权威方以保持一致性。

具体例子

微波炉说明书说按开始键转盘会旋转，但实际上不转。这时应该修正说明书（如果设计意图就是如此），而不是修微波炉。

常见误区

  • 认为“文档跟不上代码”是正常的，文档只是代码的被动追记。
  • 为了省事直接修改代码，忽略文档更新，导致规范退化。

分支隔离作为功能工作区的契约边界

完整讲解

分支隔离可以理解为软件开发中的“平行宇宙”技术。假设你正在写一个论文，主线版本是给导师看的完整稿。现在你想重新组织第三章的结构，但又怕改到一半时导师突然要看稿，这时候你就复制一份，在新副本里大胆修改。改好了再请同学帮你检查，确认没问题后才把修改合回主稿。Git 分支就是这个“副本”的技术实现。分支的关键契约是：在你明确说“合并”之前，你在分支上的任何修改（增、删、改文件、调整文档）都不会影响主线版本。这个隔离不仅是技术上的，更是契约上的——分支名（比如 `codex`）是对该分支上所有变更意图的声明，是一个时间线的标签。分支还提供了审查空间：别人可以 diff 你的分支和主线之间的差异，逐项审查后再决定是否接受。今天 ResearchPaperBase_codex 的修订发生在 `codex` 分支而非主线，就是这个原理的应用。

原理逻辑

  1. Git 分支只是轻量指针，不复制文件，创建成本极低。
  2. 合并前通过 Pull Request 审查分支差异，保证质量。
  3. 违反分支隔离（混入无关修改）会破坏契约，增加审查复杂性。

具体例子

写论文时，复制一份新副本，在新副本里重新组织第三章，改好后再合回主稿。

常见误区

  • 误以为分支是仓库的完整副本，占双倍空间。
  • 把分支当作甩锅工具，长期不合并也不删除，成为技术债务。

概念关联

关联概念关系说明
设计文档与系统行为的一致性分支隔离让文档修正的原子性和可审查性得以保证，不和其他变更混合。
commit message 的结构化与意图编码在多分支审查时，结构化消息头帮助审查者快速跳过一个分支中不相关的提交。

  2. 如何设计一个自动化策略，防止“分支长时间未合并”导致脱离主线的风险？比如设定 30 天的存活上限，超时自动提醒分支作者或被系统打标审查？
  3. 在微服务架构下，不同服务使用不同仓库，分支隔离的效力能跨仓库传递吗？如果能，用什么机制（比如 monorepo 或 meta-repo 的工具链）来保证多仓库之间的分支一致性？

commit message 的结构化与意图编码

完整讲解

想象你收到一条短信：“到了”。这条消息本身有信息量——你知道发送者到达了某地。但你不知道是谁到了、到了哪里、什么时候到的、怎么来的。如果换成：“张三：已乘地铁到达公司南门，08:52”。这个结构化消息用了 15 个汉字，编码了发送者、交通工具、位置、时间四个维度的信息。commit message 就是一种专门设计来编码“我改了什么、为什么改、怎么改”的微型结构化文本。今天看到的 `docs: fix settings in UI` 就是一个极简例子：`docs` 说明这是文档修改（不是功能代码），`fix` 说明是修正性质的（不是新增内容），`settings in UI` 说明范围是 UI 相关的设置描述。六个单词，读的人不用打开 diff 就知道：哦，这是个文档修正，影响 UI 设置部分，我可以快速扫过或者重点关注。这是信息压缩的工程实践——用最小的文本代价传输最关键的意图。

原理逻辑

  1. 消息头采用类型（作用域）：简述的三段式结构，如 `docs: fix settings in UI`。
  2. 信息压缩等同于 Kolmogorov 复杂度：好的消息头是最短的有效描述。
  3. 帮助审查者快速判断是否需要关注，减少认知开销。

具体例子

收到短信“到了” vs “张三：已乘地铁到达公司南门，08:52”，后者用结构化信息大大提升了可理解性。

常见误区

  • 认为 commit message 只是给自己看的备忘录，用模糊词如“修改”、“update”即可。
  • 认为消息越长越好，却忽略了消息头的精准度比冗长的消息体更重要。

概念关联

关联概念关系说明
设计文档与系统行为的一致性清晰的 commit message 让文档修正的意图一目了然，避免误判为功能变更。
分支隔离作为功能工作区的契约边界在多分支审查时，结构化消息头帮助审查者快速跳过一个分支中不相关的提交。

继续深入问题

  1. 如果一个仓库产生了大量琐碎提交（如反复修正拼写），有没有算法可以自动分析 commit message 来区分“有意义的分段修订”和“可以 squash 成一次的噪声提交”？
  2. 不同团队或开源项目有不同的 commit 规范（Conventional Commits、Angular、自定义），如果想让工具链通用化解析 commit message，应该设计怎样的结构化约束或 Schema？
  3. 结合 AI 工具，能不能在写 commit 时自动根据 diff 内容生成建议的结构化消息头，然后让提交者确认或修改？这种半自动化能多大程度上提升消息质量的一致性？

概念之间的关联

三个概念构成完整工程逻辑链：设计文档作为规范来源定义系统“应该怎样”，分支隔离为修正规范提供不受干扰的工作区，结构化commit message让修正意图被快速理解。三者协同，让轻量级文档修订具备了可追溯性和协作安全性。

今日最值得记住的一句话

设计文档是系统行为的规范来源而非被动记录——当文档与实现不一致时，修正文档是恢复契约，迁就代码是放弃规范。
