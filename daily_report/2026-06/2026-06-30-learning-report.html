实验配置的数据驱动设计

Data-driven Design of Experiment Configuration

把实验方案写成 JSON，模拟运行器就是可复用的反应釜。

核心要点

  • 实验配置与执行引擎解耦，策划只需修改 JSON，无需懂代码。
  • 遵循策略与机制分离原则，使得批量实验和对比变得简单。
  • 只将需要在实验中独立变化的变量参数化，避免过度暴露内部细节。

  • 利用在线聚合算法实时计算平均值、方差等，无需存储所有数据。

鱼雷弹道散布模型

Torpedo Ballistic Dispersion Model

鱼雷散布不是简单加和，而是独立误差的方差叠加——就像射箭时手抖和风偏共同决定落点。

核心要点

  • 将总散布分解为基座误差、发射误差等多个独立误差源，每个用高斯分布描述。
  • 利用方差叠加原理（独立误差源的总方差等于各方差之和）模拟随机偏移。
  • 提供独立的参数旋钮（如瞄准精度、航行稳定性），使策划能直观调整手感并量化效果。

实验配置的数据驱动设计

完整讲解

当一个游戏需要反复测试‘鱼雷散布增大 10% 后胜率会变多少’这种问题时，如果把测试场景硬写在代码里，每次改条件都要改脚本、重新编译，风险高、效率低。实验配置的数据驱动设计，就是把实验的条件——用什么地图、双方 AI 参数、鱼雷散布值等——全部写进结构化的 JSON 文件中，然后让一个通用的模拟运行器去读取这些文件、执行实验。这样一来，改变实验条件就变成了编辑文本文件，而不是改动程序逻辑。背后的核心原则是‘策略与机制分离’：策略是你想测什么（在 JSON 里），机制是怎么执行测试（在运行器里）。这种分离也让批量化实验成为可能——可以一口气加载多个 JSON，自动跑完所有实验并生成对比报告。

原理逻辑

  1. 策略与机制分离：JSON 定义实验策略，运行器实现执行机制。
  2. 数据驱动：通过结构化配置文件驱动通用引擎，实现参数化实验。
  3. 封装与抽象：只暴露必要的可调维度，隐藏内部实现细节。

具体例子

在 tiny-sea-war 的 data/simulations/experiments/ 目录下有三个 JSON 文件，如 3v3_open_sea.json。当策划想测试把某一方的鱼雷散布误差从 1.5° 降低到 1.0° 后胜率如何变化，只需打开这个 JSON 找到 torpedo_spread_error 字段改成 1.0 保存并重新运行模拟。整个过程不需要打开任何 .gd 文件或了解加载细节。

常见误区

  • 认为‘把所有可调参数都放进配置文件就是好的数据驱动’。实际上，只有那些在不同实验场景中确实需要独立变化的变量才适合参数化，否则会让配置文件臃肿并破坏封装。

伤害统计与分析管道

完整讲解

在游戏战斗系统中，每一次开火命中都会产生一个伤害数字。如果只把伤害数字显示在屏幕上然后立刻遗忘，我们就永远无法回答‘这个兵种的平均伤害是多少’‘哪种武器对战斗的贡献最大’这类问题。伤害统计与分析管道，就是专门负责抓住每一笔伤害事件，并实时计算平均伤害、最高伤害、总伤害等统计量的一套模块。它像一根水管，战斗系统是上游的‘用水端’，统计模块在下游‘接水’——上游每产生一滴水，下游就接住并汇入自己的池子，随时可以报出当前水深。技术上，它采用观察者模式：战斗逻辑在造成伤害时发出事件信号，统计管道监听这些信号，独立完成自己的计算，全程不需要战斗核心去‘通知’或‘帮助’它。这让统计管道可以像一个外挂的行车记录仪，随时装上或卸下，不影响战斗本身的运行。

原理逻辑

  1. 事件驱动与观察者模式：战斗逻辑发出伤害事件，统计模块独立计算。
  2. 在线聚合：使用少量中间变量实时更新统计量，避免存储海量原始数据。
  3. 分类统计：按伤害来源细粒度统计，为平衡分析提供维度。

具体例子

假设在一场 3v3 的海战中，我方驱逐舰发射了三枚鱼雷，分别造成 200、180、0（脱靶）点伤害，同时主炮累计造成 90 点伤害。伤害统计管道内部维护了 sum=200+180+0+90=470, count=4, max=200。它还能进一步将伤害按来源分类，比如鱼雷伤害总和 380，主炮伤害 90，于是可以回答：‘这一局鱼雷占总伤害的 80.8%’。这些数据随后会被模拟运行器收集，作为平衡报告的关键指标。

常见误区

  • 认为统计模块必须保存每一次伤害的详细记录才能算出统计量。其实，对于平均值、方差这类常用统计量，可以通过‘在线算法’（如 Welford 方法）只用少量中间变量就实现实时更新，完全不需要存储海量原始数据。

鱼雷弹道散布模型

完整讲解

真实的鱼雷发射出去后，并不会笔直地飞向准星指向的那个点，而是会受各种干扰影响，形成一个散布椭圆。在游戏里，我们不可能模拟完整流体力学，但可以用一个聪明的简化：把总散布拆成几个独立误差源的叠加。比如，鱼雷发射基座本身有微小的对准误差（基座晃动），鱼雷在水中航行时还会受到随机扰动（流场、机械振动）。每个误差源我们都用一个概率分布来描述——通常假设是钟形的高斯分布，用标准差来衡量它的‘不靠谱’程度。数学上，如果这些误差源彼此独立，那么总散布所对应的随机变量就是这几个独立随机变量之和，其方差就等于各自方差的和，标准差则是平方和开根。这样一来，游戏设计师就可以拥有两个独立旋钮：一个调‘瞄准精度’，一个调‘航行稳定性’，单独拧哪个都会改变鱼雷命中率，但改变的物理直觉和视觉表现却不同。

原理逻辑

  1. 将物理不确定性建模为独立随机变量的叠加。
  2. 独立随机变量之和的方差等于方差之和：Var(X+Y) = Var(X) + Var(Y)。
  3. 通过控制每个误差源的标准差，实现物理可信且可调的散布行为。

具体例子

设想在 tiny-sea-war 的武器配置中，一枚鱼雷的 base_dispersion 定义为标准差 σ_base = 0.6°，launch_error 为 σ_launch = 0.4°。在模拟战斗的每一炮中，系统用伪随机数生成器分别生成两个独立随机值 e1 和 e2，都服从均值为 0、标准差为对应值的高斯分布。最终的鱼雷发射角 = 理想瞄准角 + e1 + e2。这样一来，策划如果想单纯提高瞄准稳定性，只需把 base_dispersion 从 0.6 改成 0.4，而保持 launch_error 不变，那么鱼雷命中率的变化可以通过批量模拟立即量化，而不是靠人眼去看几百次射击动画。

常见误区

  • 直觉上以为误差叠加就是简单地把误差幅度相加，比如 0.5° + 0.3° = 0.8°。但独立误差的叠加遵循方差相加规则：标准差是平方和的平方根，比直接相加小。如果误用加法，会导致游戏里散布过宽，鱼雷命中率急剧下降。
