Git工作区状态的暂存区边界

Git Staging Boundary

工作区的修改就像写在草稿纸上的铅笔字——存在，但没有签名，也没有日期戳。

核心要点

  • Git的三区模型：工作区、暂存区、对象库，只有进入对象库的提交才永久记录并携带时间戳和作者信息。
  • 暂存区是提交前的意图标记区，git add将修改放入暂存区，但时间信息只有在git commit后才绑定到提交对象上。
  • 工作区修改在Git视角中是“透明的”——git status能发现文件变化，但不记录修改时间、修改者，也不提供任何审计元数据。

未提交变更的时间归属不确定性

Uncommitted Changes Time Uncertainty

工作区里的未提交修改就是没有刻印日期的陶片：它们的内容可能非常有价值，但它们的年代归属永远只能是推断。

核心要点

  • Git本身不记录工作区修改的时间信息，其对象模型中只有commit和tag携带时间戳。
  • 文件系统元数据（mtime、ctime）可被工具随意修改，作为证据的可靠性极低。
  • 仅凭git status快照无法推断修改的时间窗口，需要组合编辑器本地历史、系统审计日志等多源辅助证据才能进行粗略估计。

精读文章 · 计算机科学

Git工作区状态的暂存区边界

完整讲解

想象一下，你的电脑桌面上有三个抽屉。第一个抽屉叫“工作区”，你在这里随意写字、涂改、丢草稿纸，乱是乱一点，但这是你的自由创作空间。第二个抽屉叫“暂存区”，你想把哪些草稿正式归档，就先放进这个抽屉。第三个抽屉叫“对象库”，这是一个带编号、带日期戳、带锁的铁柜子——一旦放进去了，内容就永远保存在Git的历史里，谁也别想偷偷改。 这个比喻的核心就是Git的三区模型。工作区就是你现在在编辑器里修改代码的地方，改完了Git会发现文件变了，但它不记录这些变化。暂存区是你用git add命令把选中的修改放进来的地方——放进来了，这些修改就有了“我想提交这些”的意图标记。对象库是你git commit之后产生的那个带SHA-1哈希值的提交对象——这是Git历史中唯一的永久记录。 关键的认知是：工作区的修改在Git的视角中是“透明的”。git status能告诉你哪些文件被改了，仅此而已。它不会告诉你谁改的、什么时候改的、改了多少次——这些信息只绑定在提交对象上，不绑定在工作区文件上。这就是为什么今天我们看到tiny-sea-war有数百个文件在工作区被修改，但我们不能断定这些修改发生在今天，因为它们从未踏入暂存区和对象库的大门。 这道边界也是工程审计中一个非常重要的约束：如果你在做合规检查或事故调查，只有进入了对象库的提交才能作为“事实”来使用。工作区里的修改，哪怕内容再完整、再重要，在法律或流程意义上都只是“线索”而非“证据”。理解这道边界，你就理解了为什么敏捷开发强调小步提交——频繁的git add和git commit不仅是为了备份，更是为了给每次修改都留下可追溯的时间锚点。

原理逻辑

  1. 工作区：当前编辑的文件状态，Git检测变化但不存档。
  2. 暂存区：通过git add存入的修改集合，代表“准备提交”的意图，但仍无时间记录。
  3. 对象库：git commit生成带SHA-1哈希的提交对象，永久保存并绑定作者、时间戳等元数据。
  4. 证据资格只有进入对象库的提交才具备，工作区修改仅作线索。

具体例子

早晨上班后你用git status检查仓库，发现三个文件标记为modified。你无法回答经理的问题：“你是今天早晨改的吗？还是昨晚下班前改的忘了提交？”这些问题Git回答不了，因为答案全在工作区那道边界之外。只有执行git add . && git commit -m "fix: 修复登录页面的样式问题"之后，Git才会给这次修改打上时间戳和作者信息，这些元数据才成为可审计的事实。

常见误区

  • 一个非常普遍的误区是认为“git status显示文件被修改了，Git一定知道修改发生在什么时间”。实际上，Git内部完全不做这件事。Git的对象模型基于内容寻址——它用文件内容的SHA-1哈希值来找对象，而不是用时间戳。当你git status时，Git做的是实时比较：拿工作区文件的当前内容，计算一个哈希，然后跟暂存区或最后一次提交中对应文件的哈希比一下。如果哈希不同，就报告“已修改”。整个过程没有任何时间记录环节。文件系统层面的mtime确实存在，但可以被touch命令任意更改，Git既不读取它也不信赖它，因为Git的设计哲学是只信任自己生成的内容哈希。

关联概念关系说明

精读文章 · 计算机科学 / 软件工程

未提交变更的时间归属不确定性

完整讲解

既然我们已经知道Git不对工作区修改记录时间，那么一个直接的问题就来了：如果我现在看到一个文件在工作区被修改了，我能推断这个修改是什么时候发生的吗？答案非常微妙：能做推断，但不确定，而且推断的质量取决于你手里有多少辅助证据。 先说明为什么Git本身提供不了这个信息。Git的整个历史记录系统建立在四个对象类型上：blob（文件内容）、tree（目录结构）、commit（一次提交快照）、tag（指向某个commit的标签）。这四类对象中，只有commit和tag携带时间戳，而它们都是通过git commit或git tag主动创建的。工作区中的修改没有对应任何Git对象，因此在这个四类对象的图谱里根本不存在，自然没有时间信息。 那我们怎么办？靠工作区之外的东西。文件系统在每个文件上维护了修改时间（mtime）、状态变更时间（ctime）和访问时间（atime）——这些是操作系统记录的，但不是为版本控制设计的。它们可以被程序的写操作自然更新，也可以被人为用touch命令任意修改。编译器的编译输出、代码格式化工具的自动保存、甚至IDE的自动备份功能，都会改变这些时间戳，让它们失去作为“开发者有意识修改时间”的证据意义。 还有编辑器本地历史——VS Code、IntelliJ IDEA等现代编辑器会在本地保存文件的历史版本，这些历史记录带有时间信息。但它们是本地存储的，不上传到Git仓库，CI环境通常访问不到。操作系统级别的审计日志（如Linux的auditd）可以记录哪些进程在什么时间写入了哪些文件，但这种级别的日志通常只在安全要求极高的环境中才会开启和保存。 所以今天我们看到tiny-sea-war有数百个文件的待确认修改时，我们能确定的是：“在拍摄快照的这一刻，这些文件是这个状态”。我们无法确定这些修改发生在检查窗口（06-19 00:00 至 06-20 00:00）内——它们可能是更早积累的编辑，也可能是刚刚完成的改动，甚至可能是从其他分支合并时残留的半成品。这个不确定性把所有的待确认变更锁定在“线索”层面，而无法晋升为“事实”。这也是为什么在整个concept_relevance文件中，tiny-sea-war的所有概念都被严格标注为“待确认”——不是工程上的保守，而是证据逻辑的必然。

原理逻辑

  1. Git对象模型：只有commit/tag带时间戳，工作区修改无对象对应。
  2. 文件系统时间戳：mtime/ctime可被轻易修改，不可信赖。
  3. 辅助证据源：编辑器本地历史、系统审计日志等可辅助推断，但各有限制。
  4. 证据层级：无时间锚点的修改只能作为线索，不能作为事实。

具体例子

假设你和同事共用一台开发服务器，你发现三个文件显示modified，但你不记得改过它们。是昨晚同事改的忘了提交？是某个定时脚本自动修改？还是一个月前的实验残留？在没有频繁提交习惯的情况下，多个开发者的修改会在工作区里混在一起，再也分不清谁是谁。解决办法是频繁提交，一人一个分支，让每次修改都在Git的历史中获得明确的时间戳和作者归属。

常见误区

  • 很多人以为只要文件的时间戳（mtime）在今天之内，就可以认定修改发生在今天。这个想法有两个致命漏洞。第一，mtime可以轻易被修改——touch命令就是为了干这个的。第二，很多自动化工具会修改mtime却不代表有语义变化：代码格式化工具保存文件时可能只重新排版而不改逻辑，但mtime照更新不误；编译前的预处理脚本可能生成临时文件而后又删除，留下mtime变化却没留下内容。所以用mtime来推断修改时间，相当于用温度计来测风速——工具用错了地方，结果自然不可信。

文档编号体系与Agent可读性

完整讲解

当你在图书馆里找书时，你不需要一本一本地翻目录——你只需要看索书号，就能知道这本书属于哪个学科领域、在哪个书架的哪个位置。这就是编号体系的力量：它把人类知识的主观分类转化成了客观的空间坐标。 今天我们在tiny-sea-war项目中观察到，docs目录下的文档进行了一次大规模重命名：从前文档叫core_mechanics.md、combat_formula.md、art_design.md这样的自然语言文件名，现在变成了10_game_core_mechanics.md、12_combat_formula_design.md、40_art_direction_design.md这样的编号前缀结构。编号本身暗示了一个域分类体系：10系列是核心玩法，20系列是数据架构，30系列是技术架构，40系列是美术方向，90系列是审计文档。 这个做法对我们人类来说帮助不大——我们本来就能从文件名推断出core_mechanics是讲核心机制的，加个10_前缀不过是在我们已知的信息前加了个数字。但对程序来说，这是天壤之别。一个自动化脚本（或一个大语言模型Agent）想要“读取所有技术架构相关的文档”，如果文件名是自然语言，它必须理解什么是“技术架构”、这个词与“程序设计”“实现研究”的语义差别在哪里——这是需要人类级别语言理解的困难任务。但如果文档遵循编号体系，Agent只需要做一件事：用正则表达式匹配docs/3[0-9]_.*\.md，就能精确且无遗漏地找到所有30系列的技术架构文档。 这就是“为机器可读性而设计文档结构”的核心思想。编号前缀充当了一种命名空间，它把文档的领域归属从需要语义理解的自然语言，变成了可被简单模式匹配解析的语法规则。对LLM Agent来说，这意味着从“语义猜测”升级为“结构化查询”——Agent不再需要去猜“这篇文档讲的是不是技术架构”，而是直接根据编号前缀就可以确定。同样，AGENTS.md文件充当了知识库的使用说明书，告诉Agent编号体系怎么用、每个目录包含了什么类型的知识、在什么任务下应该查阅哪个编号段的文档。workorder/目录则可能承载Agent可执行的具体工作指令，让Agent带着明确的检索目标去消费这个结构化的知识库。

原理逻辑

  1. 编号编码领域分类：将文档按照预定义的域分类学赋予数字前缀，10-19核心玩法，20-29数据，30-39技术，40-49美术，90-99审计。
  2. 模式匹配代替语义理解：Agent通过正则表达式（如docs/3[0-9]_.*）即可精确检索，无需理解自然语言。
  3. AGENTS.md作为元说明书：指导Agent如何利用编号体系、各目录的知识类型和适用场景。
  4. 结构化知识库降低检索成本：为自动化工具设计清晰的接口，使知识消费变得机械、快速且准确。

具体例子

假设你给一个LLM Agent下达指令：“查找项目中所有关于战斗系统数值平衡的设计文档，总结其中的公式设计原则。”如果文档按自然语言命名，Agent需要理解“战斗系统”“数值平衡”等语义，逐个读取文件判断，容易遗漏或误判。但如果文档遵循编号体系，Agent可以根据AGENTS.md中的说明，用grep docs/1[2-9]_.*.md和docs/4[4-4]_.*.md瞬间排除95%的无关文档，只读取那几篇确定相关的，摘要和总结的质量自然大幅提升。

常见误区

  • 一个常见的误读是把这种编号体系理解为“只不过给文件加了数字前缀，好看一点”。这个理解漏掉了关键的设计动机：编号前缀的选择本身已经隐式编码了一个领域的分类学。如果团队的编号只是按时间顺序流水号（01、02、03...），那确实只是好看——只是把文件排了序。但tiny-sea-war的编号体系显然经过设计：10-19是核心玩法，20-29是数据，30-39是技术，这形成了一个完整的域分类体系。这种分类本身对自动化Agent来说是有语义价值的——Agent可以通过编号段来推理一个文档属于哪个领域，而这个推理不需要任何自然语言理解。

继续深入问题

  1. 如果文档编号越多越细（比如从两位数扩展到三位数甚至四位数的编号段），对Agent来说检索精度会提升，但这种复杂度会不会增加人类维护者的认知负担？这条边界应该划在哪里？

概念之间的关联

今天三个概念形成了递进的逻辑链条。C9（Git暂存的边界）建立了最基本的认识：Git只对进入暂存区并提交的内容负责，工作区的修改在Git眼中是透明的——存在，但没有元数据。这直接导致了C10（时间归属的不确定性）：当工作区有未提交修改时，我们无法从Git本身得知这些修改是什么时候做的，这构成了工程审计中的一个真实困境。C2（文档编号与Agent可读性）则展示了另一个维度的“可读性设计”：不仅Git需要结构化来保证可追溯性，文档的组织也需要结构化来保证自动化工具（包括大语言模型Agent）能够可靠地消费和理解。有意思的是，C10描述的困境恰好反向说明了C2这种实践的价值——当Git无法告诉我们某些信息时，我们更需要通过其他结构化的组织方式来降低项目知识的理解和检索成本。

今日最值得记住的一句话

工作区的修改无论有多少、多重要，只要没有git add和git commit，Git就既不记录也不保护它们——这是设计，不是bug。
