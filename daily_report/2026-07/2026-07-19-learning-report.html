自动化模拟实验的命名契约与数据可追溯性

完整讲解

当你需要为很多关卡跑自动模拟实验时，会产生一堆结果文件。如果文件名只是随便取的数字编号，每次想找“S型第2关跑了多少轮的胜率实验”都得打开文件看内容，非常麻烦。命名契约就是把实验的关键信息——比如关卡类型、关卡编号、实验种类、运行次数——直接写进文件名，按照固定的格式。这样一来，脚本用简单的文件搜索就能知道有哪些实验，核对有没有漏掉的实验也不需要打开任何一个 JSON，就像快递单上把收件人、地址、物品类型全印在外面，分拣机扫一眼就能分类。这种做法背后的原则叫“约定优于配置”，它把文件系统变成了一个不用数据库就能查询的小型信息库，让数据本身自带说明，方便自动化工具追溯每一次实验。

原理逻辑

  1. 确定关键维度：关卡类型、编号、实验类型、运行次数
  2. 设计固定格式的文件名模板，用分隔符拼接维度
  3. 代码通过模式匹配发现文件，而非解析文件内容
  4. 自动化检查缺失实验，保证完整性

具体例子

文件 data/simulations/experiments/level_s02_win_rate_20.json 中，level 代表关卡级实验，s02 是 S 型第 2 关，win_rate 表示胜率实验，20 是运行次数。任何 CI 脚本可以用 ls data/simulations/experiments/level_s02_* | wc -l 来检查 S02 的实验文件是否齐全，而不需要打开 JSON。

常见误区

  • 认为文件名随便取就行，反正自己记得住。但批量处理或自动化时，随意取名会导致无法定位。
  • 以为通过文件内容解析一样方便，忽略了模式匹配在批量操作和效率上的优势。

概念关联

关联概念关系说明

关卡目标服务的领域建模与解耦模拟实验验证新规则时，实验文件的命名同样依赖该契约，保证数据可追溯。

关卡数据类型扩展与分类增强

完整讲解

在已有的关卡家族里，已经存在 T 型（教程）关卡，今天又引入了 S 型关卡，这其实是在做“分类的扩展”。用生活中的比喻就是，你原来只卖蛋糕和面包，现在要加卖素蛋糕。素蛋糕属于蛋糕的一种，它继承蛋糕的基本属性（比如原料、尺寸），但又多了一些素食特有的约束（不用蛋奶）。在程序里，这就叫“泛化-特化”关系：关卡是泛化的基本类型，S 型关卡是它的一个特化子类型。做这件事要小心，因为已有的系统是按照老的分类设计的，解析关卡的代码默认字段就是过去那些。加入 S 型可能带着新字段，如果系统没有准备扩展点，旧代码读新数据就会崩溃。因此，这种扩展必须考虑“前向兼容”：新数据要多带字段，旧代码读到时最多忽略而不报错；同时数据模式文档也要同步更新，让人和程序都知道多了什么。

原理逻辑

  1. 在关卡分类体系中增加 S 型子类，继承通用属性，添加特有字段
  2. 更新数据模式文档记录新字段和类型
  3. 在配置注册表中声明新关卡实例
  4. 生成模拟实验文件，验证新关卡平衡性
  5. 确保旧代码读取新数据时忽略未知字段而不报错（前向兼容）

具体例子

formal_level_01.json 增加了 78 行，可能包含 S 型关卡特有的字段，比如 enemy_wave_pattern: 'swarm'，而 T 型关卡没有这个字段。配置注册表 config_registry.gd 新增了 S 型关卡的声明，告诉系统“现在有 S02、S03 这些关卡了”。模拟实验目录下也出现了 level_s02_win_rate_20.json，用于验证新关卡的平衡性。整个变更连成了一条从数据定义到注册到验证的完整链路。

常见误区

  • 误以为加新类型就是加几个 if-else 判断，导致代码分支爆炸，每次新增类型都要修改大量地方。
  • 忽略前向兼容性，旧代码遇到新字段崩溃，没有预留扩展点或做好忽略逻辑。

概念关联

关联概念关系说明
自动化模拟实验的命名契约与数据可追溯性新增的 S 型关卡必须向配置注册表声明自己的元数据，否则游戏流程无法发现和加载它；每种新关卡类型都需要对应的模拟实验文件，命名契约中的类型标识符保证了自动化工具能按类型筛选实验结果。
关卡目标服务的领域建模与解耦S 型关卡可能引入新的目标增强规则，领域服务需要根据关卡类型分发不同的判断逻辑。

  2. 如果某个关卡类型后来被废弃，应该如何在注册表中标记而不删除已有历史数据，并阻止该类型被新代码误用？
  3. 类型扩展如何与游戏引擎中的序列化/反序列化库配合，让 JSON 中的新字段自动映射到对象属性？

关卡目标服务的领域建模与解耦

完整讲解

游戏里一场战斗的流程很复杂：开始战斗、敌人一波一波出现、玩家消灭敌人、触发增援、最终胜利或失败。如果把这些“流程步骤”和“什么时候算增援、增援什么敌人”的规则全写在一个大文件里，一旦策划想调整增援条件，程序员就得在几千行代码里小心翼翼地修改，很容易动到无关的流程代码，引得 bug 四起。领域建模与解耦的做法是把这两类事情分开：battle_session.gd 只负责战斗的总体步调，它在合适的时刻会问一句“现在要不要触发增援？”，但具体怎么判断增援条件、增援什么敌人，它一概不管。这些细致的规则被封装在 level_objective_service.gd 这个专门的服务里。这样修改增援规则时，只需要更换那个服务里的逻辑，战斗流程代码纹丝不动。这种分工就是领域驱动设计中“应用服务”与“领域服务”的经典分界：应用层只管流程编排，领域层负责核心业务规则。

原理逻辑

  1. 识别业务规则与流程编排的不同职责
  2. 将业务规则抽取到领域服务，只暴露询问接口
  3. 应用服务在关键步骤调用领域服务，根据返回值决定下一步动作
  4. 领域服务独立于应用服务，可以单独单元测试
  5. 当规则变更时，只需修改领域服务内部逻辑，应用服务不受影响

具体例子

假设 S 型关卡的目标增强条件是“当前波次敌人数少于 3 个时，发射信号激活导航陷阱”。这条规则在 level_objective_service.gd 中以一个方法形式存在，比如 should_reinforce(current_enemy_count, wave_stage)。battle_session.gd 在每帧更新时会调用这个方法，如果返回 true，它就执行增援流程。当未来规则改为“若玩家处于护盾充能状态则延迟增援”，只需修改 level_objective_service.gd 中的判断逻辑，battle_session.gd 不用动。

常见误区

  • 认为把逻辑拆成多个文件是过度设计，全写在一个地方更直观。但短期直观换来的长期痛苦是规则变动时难以定位和测试。
  • 没有清晰判断什么逻辑应该放在应用层还是领域层，导致边界模糊，应用层混入业务规则，领域层又沾染流程控制。

关联概念关系说明
