Git 工作区、暂存区与提交历史的证据层级差异

完整讲解

你可以把Git想象成一个有三个抽屉的柜子。第一个抽屉叫工作区，里面放着你正在涂涂改改的草稿纸，修改随时可以发生，但别人完全看不到，也找不到任何记录——就像在纸上用铅笔乱画，擦了就没了。第二个抽屉叫暂存区，相当于你把几张觉得重要的草稿纸用回形针别起来，告诉Git“这些改动我打算记下来”，但它还没正式登记。第三个抽屉叫提交历史，就是把那片别好的纸装订成册，盖上日期章和签名，永远保存在档案柜里。从证据的角度看，草稿纸只能说明“某人可能在某时动了笔”，但无法作为可靠的工程记录；装订成册的档案才是板上钉钉的事实。今天tiny-sea-war仓库的未提交修改就属于草稿纸状态——我们能推测开发者在调游戏流程、在写测试，但不能说这些工作已经完成并交付。

原理逻辑

  1. Git 将版本控制分为三个独立区域：工作区（Working Directory）、暂存区（Staging Area / Index）和提交历史（Commit History），每个区域对应不同的文件状态和持久化程度。
  2. 工作区的修改是易失的，不纳入版本管理；暂存区通过 git add 将修改标记为预备提交；提交历史通过 git commit 将暂存内容固化，生成唯一的 SHA-1 哈希，形成不可篡改的证据。
  3. 从证据强度看：工作区 < 暂存区 < 提交历史；只有提交历史中的内容才是可追溯、可审计的工程交付。

具体例子

一位开发者修改了游戏流程文件 game_flow.gd，新增了暂停菜单状态，但忘记执行 git add 和 commit。第二天代码库中并无该功能，不幸的是临时文件被清理，工作全部丢失。

常见误区

  • 认为文件内容改变了就等于工作完成，忽略了提交的必要性。
  • 认为 git status 显示 modified 就代表当天有可交付的产出，实际上修改可能是几天前遗留的，或仅是格式调整。

概念关联

关联概念关系说明
待确认变更线索与已确认提交事实的区分原则直接支撑该原则：正因为工作区修改不是可审计事实，所以必须将工作区的变动归类为待确认线索，而不应直接当作已确认的进度。
基于远端 Git 引用指针的最小权限变更监控为监控机制提供基础：远端监控只能检测已提交的引用指针变化，不会也无法感知工作区的任何动静，这恰好遵循了证据层级的约束。

待确认变更线索与已确认提交事实的区分原则

完整讲解

理解了Git三区的证据层级之后，我们就可以把这一思想应用到日常工作记录中。当我们在写工作汇总或项目复盘时，必须清晰地区分“线索”和“事实”。工作区中未提交的修改，就像犯罪现场的脚印——它提示可能有活动发生，但你不能直接把它当成结案证据。只有经过git add和git commit，文件进入提交历史，才算有了“监控录像”般的事实。今天tiny-sea-war仓库的两个未提交文件就是典型线索：game_flow.gd改了19行，我们可以推断开发者可能在调整游戏流程；scene_presentation_test.gd新增54行，暗示场景展示测试在开发中。但这些推断前面必须加上“待确认”三个字，因为明天这些修改可能被完全推翻，也可能只是随手实验后就被丢弃。相反，public-library仓库的远端监控所检测到的ref指针变化，则是基于已提交SHA的确定事实，值得被正式记录。这种区分不仅是严谨，更是一种工程伦理：我们提供给他人做决策的信息，必须标注清楚是推测还是确认。

原理逻辑

  1. 任何未进入提交历史的代码变动，都只能作为“待确认线索”，因为它们随时可能被丢弃或推翻。
  2. 已提交的变更构成了“已确认事实”，它们拥有时间戳、作者和不可篡改的哈希链，可以作为工程决策的基础。
  3. 通过在工作记录中明确标注信息状态，可以防止将推测误当作进度，避免错误决策，同时培养团队的工程严谨性。

具体例子

一位新入职的工程师在日报中写“完成了敌方AI的行为树重构”，但代码库中并无相关提交——他只是在本地分支上写了个半成品。主管在会议上宣布后遭到质疑，最终尴尬修正。此后团队在日报模板中增设“证据状态”栏，强制区分。

常见误区

  • 以为 git status 有输出就代表有可报告的工作量，实际上这些输出可能是无意义的实验残留或自动格式化改动。
  • 在汇报时，把未提交的本地修改混入已完成工作，营造出虚假的进展感，最终导致项目状态失真。

概念关联

关联概念关系说明
Git 工作区、暂存区与提交历史的证据层级差异建立在三区证据层级之上：该模型告诉我们工作区不可靠，因此必须将其标记为线索；暂存区是意图，提交才是事实。

基于远端 Git 引用指针的最小权限变更监控

完整讲解

现在我们把目光从本地仓库投向远程协作。假如你需要监控上百个Git仓库有没有新代码提交，你肯定不会一个个克隆下来——那会占用海量的硬盘和带宽。有一种极简的办法：用git ls-remote命令，只问远端仓库一件事——“你的分支指针现在指向哪个提交哈希？”如果这个哈希值和上次记录的不同，就说明有新提交发生了；如果一样，就什么也不用做。这就像你不需要去敲邻居的门看他是否在家，只需要看一眼门口的灯亮没亮就足够了。今天的public-library仓库就采用了这种监控机制，它不下载任何代码，不接触文件内容，不依赖本地工作区，仅仅通过读取引用指针的元数据，就完成了变更感知。这种设计体现了“最小权限”和“最小知识”的思想：监控方与仓库之间只需要约定“指针当前值”这个最小接口，系统边界因此变得清晰而安全。当然，它也有明确的认知边界：只能知道指针动了，却不能区分这是一次常规合并还是一次紧急修复，更无法知道改了哪些文件。这种“准确的有限性”恰恰是一种值得学习的设计哲学——知道自己的能力边界，比用猜测填满空白更可靠。

原理逻辑

  1. Git远端仓库通过引用（refs）指向具体的提交对象，分支、标签等引用本质上是一个指向提交 SHA 的指针。
  2. git ls-remote 可以获取这些引用的当前值，无需克隆仓库，仅依赖远程仓库的读取权限，极大地减少了网络和存储开销。
  3. 监控逻辑仅需比对当前指针 SHA 与之前记录的 SHA，若不同则标记变更事件，不涉及任何内容解析，天然保证了安全隔离和最小耦合。

具体例子

public-library 仓库的监控脚本每次执行 git ls-remote origin，获取 main 分支的 SHA，与上次记录的 def456 比较发现变为 abc123，于是记录“main分支指针已变化”，但不执行任何 clone 或 fetch 操作，资源消耗极低。

常见误区

  • 认为“远端指针变化就代表有代码更新”，实际上强制推送或回滚也可能导致指针变化，而实际内容可能并未向前演进。
  • 期望这种监控能告知“改了什么文件”，但它仅提供引用指针的变化，不具备任何文件级别的感知能力。

概念关联

关联概念关系说明
Git 工作区、暂存区与提交历史的证据层级差异监控只检测已提交的 SHA 变化，完全忽略工作区的不确定状态，是对证据层级思想的最佳实践：只依赖提交历史这一最高强度的证据。
待确认变更线索与已确认提交事实的区分原则监控接收的是已确认的提交事实（指针移动），而在工作记录中也应当将远端指针变化视为已确认的变更信号，本地工作区修改则属于待确认线索。
