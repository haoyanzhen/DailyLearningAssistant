场景展示测试的模式

Scene Display Test Pattern

场景展示测试把游戏画面的正确性变成了可自动重复的检查，像视觉层面的单元测试，让设计师和程序员不再靠‘人眼跑一遍’来保证UI和特效的呈现。

核心要点

  • 不依赖截图对比，而是检查节点属性（可见性、文本、动画状态）来验证视觉正确性。
  • 像单元测试一样自动化，可集成到CI中，每次代码改动后自动运行。
  • 将直觉判断转化为客观证据，降低回归风险，提高视觉质量的稳定性。
软件工程 ★★★

地形数据的契约式建模与验证

Contract-based Terrain Data Modeling & Validation

地形数据的契约式建模用‘定义 + 自动校验 + 消费’的管道，将地图数据的质量保障从人工审查变成了可执行的规则，一条错误的坐标类型尚未进入游戏就会被拦截。

核心要点

  • 为地形数据定义严格的数据契约（字段类型、长度、约束），形成规范化的数据定义。
  • 编写校验脚本，在资源导入或提交前自动检查，将错误拦截在开发早期。
  • 构建“定义—校验—消费”的数据管道，保证只有通过校验的数据才能进入游戏运行时。
软件工程 ★★★★

游戏流程控制与配置注册表的边界分离

Separation of Game Flow Control and Config Registry

配置注册表在游戏流程和数据文件之间插入了一层抽象，让流程代码只依赖稳定的查询接口，有效隔离了数据格式变化对核心逻辑的冲击。

核心要点

  • 游戏流程通过配置注册表的抽象接口获取数据，不直接依赖文件格式或全局变量。
  • 注册表封装数据查找和转换细节，数据格式变化只影响注册表内部实现。
  • 遵循依赖倒置原则，让核心逻辑保持稳定，数据层可以独立演变，降低维护成本。

场景展示测试的模式

完整讲解

游戏不只有后台逻辑，还有直接呈现在玩家眼前的画面、文字、血条、按钮、动画效果。这些视觉元素是否出现在正确的位置、是否在正确的时机显示或隐藏，同样会严重影响游戏体验。场景展示测试就是专门为这些‘画面是否正确’而写的自动化检查。它不像常规逻辑测试那样只检查数值和状态，而是会加载一个完整的游戏场景，运行一段脚本，然后查看场景中各个节点的属性——比如某个精灵是否可见、某个UI标签的文本是否正确、动画是否播放到了指定帧。这相当于雇了一个永不疲倦的质检员，每次代码改动后都自动打开游戏界面看一遍，确保画面没坏。它的底层原理并不复杂：游戏引擎中一切可见元素都是对象，测试框架可以像检查普通变量一样检查这些对象的属性。场景展示测试的本质就是把‘看着没问题’的直觉，转化为‘属性值符合预期’的客观证据。

原理逻辑

  1. 加载完整游戏场景并运行脚本，检查场景中节点的属性（如可见性、文本、动画状态），而不是进行像素级截图对比。
  2. 利用游戏引擎的对象模型，将视觉验证转化为可自动化执行的断言，消除环境差异带来的不稳定。
  3. 将视觉正确性验证前置到开发流程中，与逻辑测试一样纳入持续集成，缩短反馈周期。

具体例子

func test_battle_ui_shows_damage_text(): var scene = load_battle_scene("test_map") simulate_hit(enemy_ship, damage=50) await next_frame() assert_true(damage_text_label.visible) assert_equal(damage_text_label.text, "-50")

常见误区

  • 很多人以为场景展示测试就是截图对比，认为只要把当前画面和一张参考图片进行像素级比对就算完成测试。实际上，截图对比在游戏中极不稳定——分辨率变化、驱动差异、操作系统渲染差异都会导致大量误报。真正的场景展示测试通常检查的是节点的属性树、对象的可见性、文本内容、动画状态等逻辑化信息，而不是像素本身。它验证的是‘意图’而非‘像素’，因此更加稳定和易于维护。

概念关联

关联概念关系说明
Godot引擎的UID机制场景展示测试需要加载场景中引用的脚本和资源，Godot引擎通过UID机制保证资源在移动或重命名后引用依然有效，因此稳定的UID是场景测试可靠运行的基础。
配置注册表的边界分离场景展示测试的输入数据（如关卡配置、地形定义）如果通过配置注册表获取，测试本身便不用硬编码文件路径，从而与配置注册表的边界分离形成呼应。

地形数据的契约式建模与验证

完整讲解

在游戏项目中，地形数据往往千变万化——岛屿形状、水深区域、生成点位置，这些都需要以数据文件（如JSON）的形式交给引擎读取。但手工编辑大量数据难免出错：字段名拼错、类型不对、坐标数组长度不匹配，任何一个低级错误都可能导致游戏崩溃或表现异常。‘契约式建模’指的是为这些数据预先设定一份严格的‘合同’（契约），明确规定每个字段是必填还是可选、它的类型是字符串还是数字、数组至少有多少个元素、不同地形块之间有什么关联约束。有了这份合同，就能编写独立的校验工具，让它像一位严苛的审计员，在数据进入游戏之前自动逐条检查，一旦发现违约立即告警。在tiny-sea-war中，地形定义文件terrain_definitions.json、校验脚本validate_terrain_definition.py和生成点构建工具build_custom_battle_spawn_slots.py恰好构成了一条‘定义—校验—消费’的数据管道，把数据建模、质量控制和运行时消费三个环节串联成一个闭环。

原理逻辑

  1. 为数据定义严格的规范契约（字段、类型、关系约束），使数据格式成为一种可引用、可检查的约定。
  2. 通过校验工具在数据使用前进行自动检查，利用规则引擎将传统的人工审查转化为机器可执行的过程，大幅提高错误发现的及时性。
  3. 形成‘定义—校验—消费’的管道架构，确保只有通过验证的数据才能被下游消费，从而在数据源头控制质量。

具体例子

{ "id": "island_01", "name": "珊瑚礁", "vertices": [[100, 200], [150, 250], [120, 300], [80, 280]], "type": "obstacle", "spawnable": false } 校验规则： 1. name字段必须存在且为长度大于0的字符串。 2. vertices数组长度必须在3到100之间。 3. 如果type为“obstacle”，则spawnable必须为false。

常见误区

  • 常见误区是认为通过校验就等于地图‘合理’。事实上，自动校验只能检查语法和结构层面的约束（如字段类型、数组长度），无法判断游戏体验层面的平衡性。比如校验程序可以确保两个生成点之间的距离是一个合法的正数，但它无法知道这个距离是否会让玩家感到不公平或者出生点过于拥挤。因此，契约式建模是质量的必要保障，但不是充分保障。

概念关联

关联概念关系说明
JSON数据模式的一致性校验地形数据的契约式建模离不开JSON数据模式的一致性校验，后者是实现前者的具体技术手段，将口头约定固化为可自动执行的约束。
关卡数据驱动设计校验通过的地形数据会被生成点槽位工具消费，体现了数据管道的生产-消费关系，而生成点配置正是关卡数据驱动设计的一环。
配置注册表的边界分离游戏流程通过配置注册表访问地形数据，而不是直接读取JSON，这意味着契约的变更只要注册表适配即可，流程代码无需感知，边界分离进一步保护了核心逻辑。

游戏流程控制与配置注册表的边界分离

完整讲解

在游戏代码中，‘游戏流程控制’指的是决定游戏从标题画面到战斗结束每一步应该做什么的逻辑，比如‘开始战斗 → 生成舰船 → 等待玩家指令 → 结算伤害 → 检查胜负条件’。‘配置注册表’则是一个集中管理所有配置数据的模块，它不参与游戏规则的决策，只负责回答‘指定地图的出生点有哪些’‘某个道具的攻击力是多少’这类数据查询。边界分离的思想是：流程代码永远通过注册表提供的抽象接口获取数据，而不是自己翻找JSON文件或直接访问全局变量。这样做的好处是，当地形定义、道具数值、UI布局等频繁变动的数据层发生格式变化时，只需要修改注册表内部的实现，所有依赖它的流程代码完全不受影响。这就像政府部门的‘窗口服务’——你去办业务只和窗口工作人员打交道，而不需要知道背后档案室换了什么管理系统。在tiny-sea-war中，game_flow.gd（应用流程）和config_registry.gd（基础设施）分属不同目录、承担不同职责，正是这种分层架构的体现。

原理逻辑

  1. 引入配置注册表作为抽象层，隐藏数据存储细节，提供统一的查询接口。
  2. 游戏流程仅依赖注册表提供的稳定接口，与具体的数据实现解耦，实现控制与配置的分离。
  3. 当数据源或格式变化时，只需修改注册表内部实现，不影响流程代码，从而保护核心业务逻辑的稳定。

具体例子

var spawn_slots = ConfigRegistry.get_spawn_points(map_id) for slot in spawn_slots: spawn_ship(slot.position, slot.faction) // 注册表内部实现 func get_spawn_points(map_id): var terrain_data = load_json("terrain_%s.json" % map_id) return SpawnBuilder.build(terrain_data) // 未来如果地形存储从JSON迁移到二进制格式，只需修改load_json这一行接口，game_flow.gd完全不必触动。

常见误区

  • 有人认为在数据很简单、项目很小的时候引入注册表层是过度设计，徒增复杂度。但实际上，游戏的数据量往往以肉眼可见的速度膨胀，一旦到了十几张地图、上百种道具时，那些直接访问JSON的流程代码就会变成难以维护的‘意大利面条’。所以这不是多此一举，而是对预期变化的前瞻性投资。当然，注册表自身的设计也需要克制，避免退化为一个包罗万象的‘上帝对象’。
