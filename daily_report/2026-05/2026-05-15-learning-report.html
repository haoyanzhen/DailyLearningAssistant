今日知识点概览

今天的三个概念分别回答三个层次的问题：模型怎样把文字变成概率输出， Agent 怎样让输入材料保持聚焦，工程需求怎样从一句愿望变成可以测试的验收条件。

Transformer 前向推理流程 从文本、分词、嵌入、注意力到词表概率，理解 LLM 一次生成的计算链路。
上下文管理 控制 Agent 看到什么、先看什么、看多少，让有限注意力服务当前任务。
可测试需求 把“要做什么”写成触发条件、边界、不变量和验收方式都清楚的工程条款。

今日知识主线

LLM 的输出来自可拆解的前向推理流水线；Agent 自动化要通过上下文管理提高输入质量； 软件工程中的需求则要写成可测试、可验收、可覆盖边界的条件。可靠不是让模型多想想， 而是让输入、过程和验收都变得清晰可控。

大语言模型 ★★★★

Transformer 前向推理流程

Transformer Forward Inference Pipeline

LLM 看起来在说话，底层是在反复把上下文转成向量计算，并预测下一个 token 的概率。

核心要点

  • 文字先经过分词，变成模型词表中的 token id。
  • Embedding 和位置编码把 token 变成带顺序感的向量表示。
  • Transformer Block 整合上下文，LM Head 输出词表上的候选分数。

小例子

输入“Transformer 的核心机制是”，模型会让“注意力”“Self”“多头”等候选获得不同概率。

记忆句：生成不是一次写完，而是一轮轮预测下一个 token。
Agent 工程 ★★★★

上下文管理

Context Management

给 Agent 的材料不是越多越好，关键是让它在当前任务中看到正确、相关、有优先级的信息。

核心要点

  • 容量管理决定哪些材料值得进入有限上下文窗口。
  • 相关性、顺序和结构会影响模型怎样使用信息。
  • 复杂任务应分阶段交接稳定产物，减少噪声和过期结论。

小例子

生成知识讲解时，提供今天的概念文件和月度记录，比塞入所有历史日报更稳定。

记忆句：好上下文不是更多材料，而是当前任务最该看的材料。
软件工程 ★★★★

可测试需求

Testable Requirements

好需求不只说明目标，还要说明什么条件下发生、结果是什么、边界怎样处理、如何验收。

核心要点

  • 触发条件、预期结果和边界条件要能被观察和复现。
  • 不变量要写清楚，避免高风险操作破坏共享数据。
  • 验收方式可以直接转成测试用例、检查清单和审阅标准。

小例子

“安全删除项目”太模糊；写清软删除、二次确认、资产范围、幂等和失败重试才可测试。

记忆句：可测试需求，是给抽象形容词装上尺子。

精读文章 · 大语言模型

Transformer 前向推理流程

完整讲解

可以把大语言模型想象成一条精密的翻译流水线。它翻译的不是中文到英文， 而是把人能读懂的文字一步步翻译成机器能计算的数字，再把计算结果翻译回下一个最可能出现的 token。

假设输入“人工智能正在改变”，模型并不是一次性理解整句话然后写完整篇文章。 更准确地说，它要回答一个局部问题：在已有上下文后面，下一个 token 可能是什么。

这条流程通常从文本输入开始。神经网络不能直接处理文字，所以分词器先把文本切成 token。 BPE 分词会从小单位出发，不断合并语料中常见的相邻片段，形成既能表示常见词、 又能覆盖陌生词的词表。每个 token 对应一个整数 id，但 id 本身没有语义， Embedding 层会把这些 id 查表成稠密向量，像是在高维语义空间中给 token 找到坐标。

接着模型加入位置信息。注意力机制擅长判断谁和谁相关，但它不天然知道序列顺序。 位置编码就像给每个 token 的向量加上座位号，让模型知道第一个、第二个和最后一个位置并不相同。 然后，多层 Transformer Block 会通过 Self-Attention 和前馈网络反复加工这些向量。 注意力让每个 token 根据上下文重新组织信息，前馈网络则在每个位置上继续做非线性变换。

最后一层隐藏向量会进入 LM Head。假设词表有十万个 token，LM Head 就会给十万个候选 token 各自打分，这些分数叫 logits。logits 再通过 softmax 或采样策略变成概率分布。 选中的 token 会接回原上下文，进入下一轮推理，这就是自回归生成。

原理逻辑

  1. 文字先被转换成 token id，让离散文本进入模型词表。
  2. Embedding 把 token id 变成可计算向量，位置编码补上顺序信息。
  3. Transformer Block 通过注意力和前馈网络整合上下文语义。
  4. LM Head 把隐藏状态映射为词表分数，再形成下一个 token 的概率分布。
文字 -> token id -> 向量 -> 带位置的向量 -> 上下文加工 -> 词表分数 -> 下一个 token 概率

具体例子

输入“今天我们学习 Transformer，它的核心机制是”，模型会先分词、嵌入、加入位置， 再经过多层 Transformer Block。最后，LM Head 可能让“注意力”“Self”“多头”“编码器”等候选获得较高分数。

候选 token可能原因
注意力前文出现 Transformer，常见搭配是核心机制是注意力。
Self英文材料中常接 Self-Attention。
多头Transformer 常见术语包含多头注意力。
编码器如果上下文偏原始论文介绍，可能讨论编码器/解码器结构。

常见误区

  • 误以为模型一次性生成整篇文章。实际生成通常是一轮轮预测 token。
  • 误以为 token 等于汉字或英文单词。token 可能是字、词、子词、标点或字节片段。
  • 误以为 Embedding 的每个维度都能直接解释成人类标签。它更像统计关系压缩出的坐标。

概念关联

关联概念关系说明
BPE 分词前向流程入口，把文字转成 token id。
Token Embedding把离散 id 变成神经网络可计算的向量。
位置编码与 RoPE给序列加入顺序信息，使注意力利用位置关系。
Self-Attention让 token 根据上下文重新组织表示。
LM Head把隐藏状态映射为整个词表上的候选分数。

精读文章 · Agent 工程

上下文管理

完整讲解

想象你正在准备考试。老师给你一本重点讲义，你可能学得很快； 如果老师把整个图书馆都搬到桌上，并说答案都在里面，你反而会失去方向。 Agent 也是这样。能处理长上下文，不等于把所有材料都塞进去就会更好。

上下文越长，越可能混入旧信息、无关细节、相互冲突的要求和过期结论。 模型在生成时会根据上下文预测后续内容，如果重要信息被埋得太深，或者噪声材料太多， 输出就可能跑偏。上下文管理就是控制输入材料：哪些要给，哪些暂时不给；先给什么， 后给什么；哪些是硬约束，哪些只是参考；多 Agent 协作时，每个 Agent 应该看到怎样的材料。

它的目标不是让模型知道一切，而是让模型在当前任务中知道足够正确、足够相关、 足够有优先级的信息。今天材料里，上下文管理同时出现在 Agent 协作和 LLM 架构教学中： 前者关注怎样给 Agent 聚焦输入，后者关注模型怎样在上下文窗口中通过注意力使用信息。

原理逻辑

  1. 容量管理：模型上下文窗口有限，要决定当前任务最需要哪些片段。
  2. 相关性管理：直接相关材料优先进入上下文，背景材料可以摘要化，无关材料应剔除。
  3. 顺序与结构管理：清楚分区任务目标、硬约束、输入材料和输出格式。
  4. 阶段管理：复杂任务拆成多个阶段，每个阶段接收上一个阶段的稳定产物。

具体例子

任务是让 Agent 根据当天材料生成知识讲解。

差的做法是把所有仓库文件、所有日报、所有旧 prompt、所有历史聊天一次性塞给 Agent， 然后只说“写今天的知识讲解”。它可能不知道哪些文件才是今天输入，可能重复讲昨天的概念， 也可能引用过期信息。

好的做法是提供今天的 concept_relevance.md、当月 knowledge_log、 当前任务 prompt，以及必要时一篇风格参考。这样 Agent 的注意力会集中在今日概念、 历史避重和指定格式三个关键点上。

常见误区

  • 误以为上下文越长，结果越好。长上下文同时提供证据和噪声。
  • 误以为只要写了约束，模型一定会遵守。重要约束应该清晰、靠前、结构化。
  • 误以为上下文管理只是 prompt 技巧。它还涉及文件组织、阶段产物、检索策略和失败恢复。

概念关联

关联概念关系说明
上下文窗口上下文管理首先受模型可处理长度限制影响。
注意力机制注意力决定模型如何分配关注，上下文管理决定材料如何进入这个范围。
渐进披露把复杂信息分阶段给出，是上下文管理的重要策略。
多 Agent 串行审阅每个 Agent 使用聚焦上下文，可以分别检查不同风险。
Prompt Engineering 中的上下文裁剪裁剪是上下文管理在 prompt 层面的具体操作。

精读文章 · 软件工程

可测试需求

完整讲解

“系统应该安全地删除 Project”听起来正确，但几乎没法测试。什么叫安全？ 删哪些？谁能删？失败怎么办？相比之下，如果需求写明先软删除、项目所有者二次确认后才清理私有资产、 不得删除全局 papers 记录、不得影响其他项目、重复触发应保持结果一致，开发和测试就能据此行动。

可测试需求的核心思想是：需求不仅要表达目标，还要表达可观察、可判断、可复现的验收条件。 它要让开发、测试、产品和 AI 编码 Agent 都能回答同一个问题：做到什么程度才算完成。

一个需求要变得可测试，通常需要补齐触发条件、预期结果、边界条件、不变量和验收方式。 触发条件说明什么时候发生、谁触发、前置状态是什么；预期结果说明系统应该产生哪些变化； 边界条件处理重复请求、权限不足、部分资产不存在、网络失败等情况；不变量说明绝对不能破坏的东西； 验收方式则把需求落到单元测试、集成测试、权限测试、幂等测试、日志检查或人工清单上。

今天材料中的发布前置校验、失败分类与重试策略、软删除与二次确认、幂等操作， 都属于可测试需求的好朋友。它们共同把一句宽泛的“系统要可靠”，拆成可执行、可观察、 可验证的工程条款。

原理逻辑

  1. 把模糊形容词改写成可观察行为，例如快、稳定、安全分别对应时间、状态和边界。
  2. 把成功路径和失败路径同时写清，避免实现只覆盖最顺的情况。
  3. 把不可破坏的不变量显式列出，让高风险操作有清晰保护范围。
  4. 把验收方式前置到需求阶段，让需求直接服务测试和审阅。

具体例子

Project 私人资产清理可以写成一组可验证要求：

测试点可验证要求
权限只有 Project 所有者或授权管理员可以发起清理。
资产范围清理 PDF、解析文本、向量索引、图谱文件、运行历史和临时文件。
保护范围不删除全局 papers 身份记录，不影响其他 Project 的派生资产。
二次确认用户确认影响范围后才执行不可逆清理。
软删除默认先把 Project 标记为软删除，支持恢复窗口或审计。
幂等同一清理请求重复执行，不应重复报错或误删额外数据。
部分失败某个资产删除失败时记录错误，并允许安全重试。

常见误区

  • 误以为需求写得越短越高级。简洁很好，但不能牺牲可验证性。
  • 误以为测试是测试工程师后面再想的事。可测试性应该在需求阶段就设计进去。
  • 误以为只测试成功路径就够。复杂系统的风险常在权限、网络、重复提交和部分失败中。

概念关联

关联概念关系说明
失败分类与重试策略可测试需求要说明不同失败类型如何处理。
幂等操作重复执行是否安全，是后台流程需求的重要验收点。
软删除与二次确认把高风险操作拆成可恢复、可确认的阶段。
全局实体与私有派生资产分离为清理需求提供数据边界和不变量。
发布前置校验把发布成功拆成多个可测试前置条件。

概念之间的关联

Transformer 前向推理流程说明模型输出如何由输入计算而来；上下文管理说明既然输入会影响输出， Agent 就要认真选择和组织材料；可测试需求则说明工程目标不能停留在愿望，而要能被检查和验收。 三者串起来，就是“模型如何计算 → Agent 如何接收上下文 → 工程目标如何被验证”。

可靠的 AI 工作流不是让模型多想想，而是让输入、过程和验收都变得清晰可控。
