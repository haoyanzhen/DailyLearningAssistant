上游产物完整性校验

Upstream Artifact Integrity Check

上游产物完整性校验是在下游任务开工前，确认输入文件存在、非空且可读取。

核心要点

  • 存在、非空、可读取是最基础的三项输入门槛。
  • 校验应发生在生成、发布、写入等下游动作之前。
  • 它把坏原料挡在生产线外，减少编造和误发布。

小例子

每天需要 5 个 work_summary_*.md 文件；如果其中一个是空文件， 下游 AI 就不应该继续据此生成概念总结。

记忆句：自动化开工前，先验材料是不是合格。
LLM 工作流 ★★★

输入缺失时停止生成

Stop Generation on Missing Input

输入缺失时停止生成，是在材料不全时主动终止正式输出，而不是用占位或猜测填坑。

核心要点

  • 停止生成是一种受控状态，不等于程序失控崩溃。
  • 它保护下游任务，不让错误输出被当成真实成果。
  • 停止时应记录清楚原因，让后续恢复有线索。

小例子

如果只检测到 4 个上游总结文件，任务应返回失败状态并说明缺失项， 而不是发布一篇“今日无更新”的正式日报。

记忆句：材料不全时，停下来比编下去更可靠。

精读文章 · 软件工程

上游产物完整性校验

完整讲解

想象你要做一道番茄炒蛋。如果番茄是烂的、鸡蛋是臭的， 或者冰箱里根本没有食材，你不会硬做，而是会先检查食材是否存在、是否新鲜。 自动化流水线也是一样：下游任务不能盲目开工，必须先确认上游产出的文件 存在、非空、可读取。

“存在”说明文件没有被删除，路径也没有写错；“非空”说明文件不是 0 字节空文件； “可读取”说明当前自动化进程有权限打开它。三项检查合起来， 就像工厂门口的质检员，把坏原料挡在生产线之外。

原理逻辑

  1. 先定位下游任务依赖哪些上游文件。
  2. 逐个检查文件是否存在，避免路径错误或产物缺失。
  3. 检查文件大小和内容，避免空文件被误当作有效输入。
  4. 确认当前用户可读取文件，避免权限问题在后续阶段才爆出来。
  5. 全部通过后再进入生成、发布或同步等正式动作。

具体例子

本地流水线每天会读取多个 work_summary_*.md 文件。 如果其中一个因为 Git 冲突或脚本中断变成空文件，没有完整性校验的话， 下游 AI 可能会对着空输入“思考”，再编造出看似合理但无法追溯的内容。 

if not exists("work_summary_AInote.md"):
    raise Error("上游文件缺失")
if size("work_summary_AInote.md") == 0:
    raise Error("上游文件为空")
if not readable("work_summary_AInote.md"):
    raise Error("上游文件不可读")

常见误区

  • 误以为只要文件存在就够了；空文件和不可读文件同样会污染下游。
  • 误以为校验可以放在生成失败后再补；前置校验能更早阻断错误扩散。
  • 误以为完整性校验只适合代码文件；日报、日志、manifest 也需要它。

概念关联

关联概念关系说明
防护式自动化完整性校验让自动化先判断输入是否可信，再决定是否继续。
流水线前置条件存在、非空、可读取就是执行后续任务前必须满足的条件。
输入缺失时停止生成校验失败后，下一步就是停止正式生成并记录原因。

  2. 大型分布式系统中，如何高效校验成千上万个上游产物？
  3. 如果文件部分损坏，例如最后几行丢失，完整性校验如何发现？

精读文章 · LLM 工作流

输入缺失时停止生成

完整讲解

校验发现问题之后，自动化系统要做选择：继续生成一个“默认结果”， 还是停止正式输出。在日报链路里，更可靠的选择是直接停止生成， 而不是写一篇“今日无更新”或“材料暂缺”的正式日报去填坑。

原因很简单：下游 Agent 会把任何产物都当成真实成果继续加工。 如果上游缺失时生成了占位内容，发布阶段可能会把占位内容发布成正式页面， 读者也会以为系统正常运行。停止生成意味着：要么输出高质量的真内容， 要么不输出正式产物，让更高层的人或机制介入。

这种停止不是程序失控崩溃，而是一种受控状态。它应该记录缺了什么、 哪个检查失败、为什么跳过后续任务，并保护已有的历史产物不被错误覆盖。

原理逻辑

  1. 先用完整性校验判断输入是否满足最低条件。
  2. 发现缺失或不可用后，立刻阻断正式生成动作。
  3. 输出标准化失败原因，让后续自动化和人都能读懂。
  4. 不更新日报、manifest 或远端同步状态，避免制造“发布成功”的假象。

具体例子

如果任务需要 5 个 work_summary 文件，但只检测到 4 个， Agent 应直接返回失败状态，并输出“上游产物不完整，跳过今日知识提炼”。 它不应该临时写一个简短总结，更不应该覆盖下游的 concept_relevance.md 或正式日报。

常见误区

  • 误以为停止生成就是程序崩溃；受控停止是一种明确的工程状态。
  • 误以为占位输出更友好；正式占位很容易掩盖真正的上游故障。
  • 误以为部分缺失时总能降级；是否降级必须由任务契约明确允许。

概念关联

关联概念关系说明
上游产物完整性校验停止生成是校验失败后的后续动作。
防幻觉生成边界输入不足时停止，是落实“不编造”的具体策略。
结构化输出契约停止时也应输出标准化错误状态，而不是随意文本。

精读文章 · 知识管理

概念图谱

完整讲解

我们经常看到各种知识清单：番茄、鸡蛋、盐、葱花。 清单告诉你有哪些东西，却没有告诉你它们之间怎么配合。 概念图谱就是把孤立概念变成关系网：节点表示概念， 边表示概念之间的依赖、类比、耦合、共同目标或延伸。

图谱比清单更适合学习，因为人理解新知识时，往往是在已有概念网络上生长新节点和新边。 当你把“上游产物完整性校验”和“输入缺失时停止生成”连成一条 “校验失败触发停止”的边，就不只是记住两个术语，而是理解了流水线可靠性的一段关键逻辑。

在 LLM 工作流里，概念图谱也能指导下游生成。高质量 few-shot 示例可以要求模型输出 [概念A] --关系类型--> [概念B] 这样的结构， 让结果从平面术语清单升级为能解释、能追踪、能扩展的知识网络。

原理逻辑

  1. 先识别文本中的关键概念，把它们作为图谱节点。
  2. 再判断概念之间的关系类型，例如依赖、触发、共同目标、类比或延伸。
  3. 为每条边写出明确语义，避免只写“相关”这种模糊关系。
  4. 最后用图谱帮助人选择学习顺序，也帮助下游 Agent 按关系展开讲解。

具体例子

今天的一小段图谱可以写成： 

[上游产物完整性校验] --失败触发--> [输入缺失时停止生成]
                                         |
                                         +--共同服务--> [防幻觉生成边界]

这比单独记住三个词更有用，因为它直接说明了校验、停止和防幻觉之间的逻辑链条。

常见误区

  • 误以为概念图谱就是思维导图；图谱的关键是每条边有明确语义。
  • 误以为“有关”就是关系；没有关系类型的边，很难支持推理。
  • 误以为图谱越复杂越好；好的图谱应服务于理解，而不是堆满连线。

概念关联

关联概念关系说明
few-shot 示例质量控制示例能引导 LLM 输出关系图，而不是平面清单。
知识日志归档日志归档的不只是术语，也可以是长期积累的概念关系。
多阶段自动化流水线图谱让下游讲解按关系展开，减少随机排列。

继续深入问题

  1. 如何自动从文本中提取概念和关系，构建知识图谱？
  2. 不同关系类型的权重和优先级应该如何设计？
  3. 概念图谱如何与推荐系统结合，辅助个性化学习路径？

概念之间的关联

上游产物完整性校验负责回答“输入材料能不能用”；输入缺失时停止生成负责回答 “材料不能用时该不该继续”；概念图谱则回答“材料可靠以后，知识应该如何组织”。 三者形成从检查到阻断再到结构化学习的递进链。

今日最值得记住的一句话

可靠的自动化不是永远往前跑，而是知道何时检查、何时停下、何时把知识连成网。
