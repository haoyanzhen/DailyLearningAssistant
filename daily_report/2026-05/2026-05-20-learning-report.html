把 LLM 自动化变成可信工程

今日知识点概览

今天的三个概念共同回答一个问题：如何让 LLM 在自动化流水线里既能生成内容， 又能保持可解析、可检查、可追溯，不把创造力变成系统风险。

结构化输出契约 提前约定字段、格式和校验规则，让 LLM 输出像 API 一样可被程序消费。
状态校验 在关键动作前检查网络、认证、远端和分支关系，确认安全后再执行。
防幻觉生成边界 限制模型只基于真实输入生成内容，证据不足时不猜测、不补写。

今日知识主线

可靠的 AI 自动化不是把 LLM 放进脚本里就结束了。它需要先规定输出契约， 让结果稳定可解析；再通过状态校验确认动作条件成立；最后用防幻觉边界约束内容来源。 这条主线把 LLM 从“会写东西的模型”推进到“能被工程系统信任的协作者”。

LLM 工程 ★★

结构化输出契约

Structured Output Contract

结构化输出契约是提前和 LLM 约定字段、类型和校验规则，让生成结果能稳定进入程序流程。

核心要点

  • 契约规定字段、格式、类型和必要的富文本结构。
  • 脚本要解析和校验输出，不合格时拒绝使用。
  • 它让 LLM 输出从自由文本变成可消费的数据接口。

小例子

知识讲解任务要求返回包含 concept_name、difficulty 和 explanation 的 JSON；缺字段或 HTML 结构不合格，就不能写入日志。

记忆句：先约定输出形状，再谈自动处理。
自动化运维 ★★★

状态校验

State Validation

状态校验是在执行关键动作前自检网络、认证、分支和远端关系，确认条件安全再继续。

核心要点

  • 把“应该没问题”的隐式假设变成显式检查。
  • 关键发布动作前要检查 DNS、SSH、远端和分支状态。
  • 校验失败时应停止并报告原因，而不是硬推下一步。

小例子

自动 git push 前先解析 GitHub DNS、验证 SSH 认证、 读取远端分支，并确认本地没有落后于远端。

记忆句：关键动作前，先确认自己站在正确状态上。
Agent 工作流 ★★★★

防幻觉生成边界

Anti-Hallucination Boundary

防幻觉生成边界给 LLM 划定信息来源范围，只允许它根据已提供的真实输入生成内容。

核心要点

  • 输入证据边界决定哪些材料可以被使用。
  • 输出约束边界决定证据不足时要停止或降级。
  • 风格可以灵活，但事实不能越过来源边界。

小例子

没有天气数据时，邮件不能写“今天气温 24°C”；它只能说明天气信息暂缺， 或者完全不提天气。

记忆句：能证实的才生成，不能证实的就留白。

精读文章 · LLM 工程

结构化输出契约

完整讲解

想象你请一个朋友帮你写摘要，但朋友每次给你的格式都不一样： 有时是列表，有时是散文，有时还画了图。你要把这些摘要自动合并到表格里， 就会很头疼。结构化输出契约就是提前和 LLM 说好： 请按指定格式返回结果，字段名、类型，甚至表格里的行列结构都要遵守约定。

这个契约有两层。第一层是格式契约，例如要求模型返回 JSON， 包含 title、date、summary 等字段，甚至要求某个字段里包含指定 HTML 结构。第二层是校验契约： LLM 返回后，脚本会解析 JSON，检查字段是否存在、类型是否正确、 富文本结构是否符合要求。

只有当输出符合契约，系统才继续使用它。否则就拒绝写入文件。 这样做的重点不是束缚模型，而是让自动化链路能稳定地接住模型输出。

原理逻辑

  1. 先定义输出 schema，让字段和数据类型稳定下来。
  2. 再要求模型按 schema 返回，而不是自由发挥整篇文本。
  3. 随后由脚本解析和校验，不符合约定就阻断下游写入。
  4. 最后把合格结果交给日志、日报、manifest 等后续阶段消费。

具体例子

在知识讲解任务中，脚本向外部 LLM 发送 prompt 时，可以要求返回： concept_name、difficulty、explanation 等字段。如果模型没有返回 explanation，或者表格缺少约定的结构， 脚本就抛出校验错误，不让不合格内容进入知识日志。

常见误区

  • 误以为只要提示词写了格式要求，输出就一定可靠；还需要程序校验。
  • 误以为 schema 越复杂越好。过度复杂会让模型更容易失败。
  • 误以为结构化会降低表达质量。好的契约约束结构，不必牺牲内容深度。

概念关联

关联概念关系说明
LLM 输出校验契约是约定，校验是把约定变成强制门槛。
配置驱动自动化schema 可以通过配置传入，让同一脚本适配不同任务。
状态校验输出契约检查内容状态，状态校验检查运行环境状态。
防幻觉生成边界契约可以规定证据字段，减少无来源扩写进入系统。

精读文章 · 自动化运维

状态校验

完整讲解

状态校验就是让自动化系统在做关键动作前，先自检一遍， 确认自己处在正确状态下再行动。它的核心是把隐式假设变成显式检查。 比如自动推送代码前，我们常常默认网络可用、SSH 密钥有效、远端可读、 当前分支正确、本地没有落后于远端。

如果这些假设没有被检查，一旦某个条件不成立，程序就可能报错、 半途停止，甚至在错误状态下继续执行。状态校验让自动化流程先问自己： 我现在能不能安全地做这件事？

对日报发布来说，这尤其重要。生成 HTML、更新 manifest、提交 git commit 都只是本地完成；真正让 GitHub Pages 更新，还需要远端同步成功。 因此远端同步前的 DNS、SSH、远端读取和分支 ahead/behind 检查， 都是发布动作的一部分。

原理逻辑

  1. 列出关键动作依赖的前置条件。
  2. 把每个前置条件写成可执行、可观察的检查。
  3. 按网络、认证、远端、分支等层次逐步排查。
  4. 检查失败时停止动作，并输出明确的失败阶段和恢复建议。

具体例子

自动 git push 前可以先执行 DNS 解析检查， 再执行 ssh -T -p 443 git@ssh.github.com 验证身份， 然后用 git ls-remote origin refs/heads/main 确认远端可读。 如果本地分支落后于远端，就停止推送并提示先同步。

常见误区

  • 误以为失败后再看报错就够了。前置检查能更早、更清楚地定位问题。
  • 误以为单个检查能代表全部状态。DNS 可用不等于 SSH 认证可用。
  • 误以为自动化越“勇敢”越好。可靠自动化应该在状态不安全时刹车。

概念关联

关联概念关系说明
防护式自动化状态校验为防护式自动化提供继续或停止的依据。
发布前置同步ahead/behind 检查决定是否能直接推送。
结构化输出契约两者都把隐含规则显式化，只是作用对象不同。
失败状态产品化校验失败后，需要把原因和恢复路径讲清楚。

精读文章 · Agent 工作流

防幻觉生成边界

完整讲解

LLM 很擅长续写。它不知道答案时，也可能根据上下文猜出一段看起来很像真的内容。 在闲聊里，这种能力常常显得自然；但在自动化流水线里， 缺少依据的生成会伤害整个系统的可信度。

防幻觉生成边界就是主动给模型画一个圈：只允许使用圈内的信息来源， 圈外的不许编。这个圈包含输入证据边界和输出约束边界。 输入证据边界规定哪些摘要、事实和数据可以进入输出； 输出约束边界规定证据不足时应写“无相关信息”、记录失败状态，或直接停止生成。

在学习日报系统里，这个原则意味着：知识讲解任务如果调用 LLM 失败， 主控流程不能自己补写一篇看似完整的内容；邮件推送可以有亲切风格， 但不能猜测收件人的现实状态，也不能编造未提供的数据。

原理逻辑

  1. 先明确哪些文件、摘要和数据是允许使用的输入。
  2. 再规定哪些内容必须有来源支撑，不能自由推断。
  3. 遇到证据缺失时，使用空结果、失败记录或降级说明。
  4. 允许风格表达，但不允许风格突破事实边界。

具体例子

如果今天没有实时天气数据，LLM 不应该在邮件里写“今天气温 24°C， 适合户外运动”。这句话看起来温和，但它没有来源。 更可靠的做法是完全不提天气，或者明确写“天气信息暂缺”。

常见误区

  • 误以为用户看不出来就没关系。事实错误会持续消耗系统信任。
  • 误以为防幻觉只是提示词问题。文件检查、来源记录和降级策略同样重要。
  • 误以为越详细越像高质量。无证据的细节越多，风险越大。

概念关联

关联概念关系说明
数据血缘血缘帮助系统判断哪些信息属于可信来源。
Prompt 风格档案风格可以变化，但事实边界必须固定。
结构化输出契约契约可以要求模型标明来源或在证据不足时输出固定状态。
状态校验生成前检查输入状态，是防止无来源扩写的第一道门。
