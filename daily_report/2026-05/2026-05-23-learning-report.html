今日知识点概览

今天的三个知识点都在处理复杂协作中的“边界问题”：输入契约守住流水线入口， 分层架构守住系统职责边界，领域不变量守住业务规则的底线。

输入契约 约定下游开工前必须拿到什么输入，以及输入必须满足哪些硬条件。
分层架构 按职责拆分系统层次，让每层只做自己的事，并通过稳定接口协作。
领域不变量 把业务中永远必须成立的规则放进核心模型，防止系统演化时越过底线。

今日知识主线

多仓库、多 Agent、多阶段流水线协作时，最危险的不是某一步出错， 而是错误悄悄流到下一步。输入契约先确认文件交接可靠；分层架构再把展示、 编排、领域、智能服务和基础设施放在合适位置；领域不变量则固定业务世界里不能被破坏的规则。 三者连起来，就是在不同边界上设置检查点。

软件工程 ★★

输入契约

Input Contract

输入契约是下游任务对上游产物的硬性约定：文件要存在、非空、可读取，并且范围要完整。

核心要点

  • 它规定“下游开始前必须拿到什么”。
  • 契约要能被代码自动检查，而不只是写在文档里。
  • 入口处拦住坏材料，能防止错误沿流水线扩散。

小例子

当每日输入范围从 5 个仓库扩展到 6 个仓库时，阶段 2 就必须确认 6 个 work_summary_*.md 都存在且可读。

记忆句：契约不是提醒别人小心，而是让系统自己知道何时不能开工。
软件工程 ★★★★

分层架构

Layered Architecture

分层架构按职责拆分系统，每层只做自己该做的事，并通过稳定接口与相邻层协作。

核心要点

  • 真正的分层重点是依赖方向，而不只是目录结构。
  • UI、API、用例、领域、智能服务和基础设施各有职责。
  • 隔离变化后，替换数据库或登录方式时不会牵动所有层。

小例子

UI 不直接操作数据库，而是通过 API、应用用例和领域核心完成业务动作， 基础设施变化也不应反向污染领域规则。

记忆句：分层的价值不是把文件夹排整齐，而是让变化停在该停的地方。
计算机科学 ★★★★

领域不变量

Domain Invariant

领域不变量是业务世界中无论流程怎么变都必须永远成立的规则。

核心要点

  • 它表达业务底线，而不是某个接口的一次性校验。
  • 应放在 Domain Core 中，让所有入口都绕不过核心规则。
  • 典型例子是 Owner 必须始终拥有自己 Project 的全部权限。

小例子

如果 Owner 可以失去删除权限，Project 可能变成无法清理的孤儿资源； 因此权限模型要用代码阻止这种状态出现。

记忆句：领域不变量就是业务世界的承重墙，不能随手打洞。

精读文章 · 软件工程

输入契约

完整讲解

想象你和朋友接力做一道菜：你负责洗菜切菜，朋友负责下锅翻炒。 如果你没有约定菜放在哪里、是否已经洗好、是否已经切好，朋友就可能找不到材料， 或者误以为你还没有完成。这个约定就是协作中的契约。

在软件系统里，输入契约就是下游任务对上游产物必须满足的硬性约定。 它不只说“需要一些输入”，而是清楚规定输入的范围、位置、格式和最低状态。 对每日学习自动化来说，当输入范围扩展到 6 个仓库后，阶段 2 必须拿到 6 个 work_summary_*.md 文件，而且每个文件都要存在、非空、可读取。

这个约束看起来严格，其实是在保护后续链路。下游一旦基于坏材料继续生成， 错误会变成概念文件、知识讲解、学习日报，最后被发布成正式页面。 输入契约的价值，就是在入口处把问题拦住。

原理逻辑

  1. 先定义下游任务依赖的完整输入集合。
  2. 再为每个输入定义最低可用条件，例如存在、非空、可读取。
  3. 在正式处理前自动执行检查，所有输入通过后才继续。
  4. 一旦检查失败，停止下游生成，并记录清楚失败原因。

具体例子

在多阶段学习流水线中，阶段 2 可以先读取配置文件，得到需要检查的仓库列表。 然后遍历列表，对每个预期文件执行三项判断： 

for repo in required_repositories:
    path = f"work_summary_{repo}.md"
    assert exists(path)
    assert readable(path)
    assert non_empty(path)

只有全部通过，才进入概念提炼。少一个仓库的总结，系统就不应该假装材料齐全。

常见误区

  • 误以为文件存在就够了；空文件和不可读文件同样会制造坏输入。
  • 误以为契约写在文档里就行；契约必须能被程序执行，否则只是提醒。
  • 误以为漏一个输入影响不大；下游内容会把缺口包装成看似完整的结论。

概念关联

关联概念关系说明
前置完整性校验输入契约规定要检查什么，完整性校验负责执行这些检查。
数据谱系输入契约保证来源材料完整，后续日报的来源追踪才可信。
领域不变量二者都是契约思想的体现，只是一个守文件边界，一个守业务边界。

精读文章 · 软件工程

分层架构

完整讲解

如果把图书馆的借书台、书架、管理员休息区和电脑系统全堆在一个房间里， 读者找不到书，管理员也很难维护。更合理的做法是按职责划分区域： 借还区负责服务读者，阅览区负责阅读，书库负责保存藏书，设备间负责基础设施。

分层架构也是这个思路：按职责把系统划分成若干层，每层只做自己该做的事， 并通过稳定接口与上下层协作。今天的论文系统设计把整体分成 UI、API Boundary、 Application Use Cases、Domain Core、Agent & Knowledge Services、Infrastructure 六层。

关键不在于文件夹名字，而在于依赖方向。UI 负责展示和输入收集，不直接操作数据库； API 边界负责把外部请求变成命令和查询；应用用例编排业务流程；领域核心表达业务对象和规则； 智能与知识服务承载 Graph-RAG 等能力；基础设施处理数据库、消息队列和外部系统。

原理逻辑

  1. 先识别系统中变化频率和职责不同的部分。
  2. 把展示、边界、流程编排、业务核心、智能服务和基础设施拆开。
  3. 规定依赖方向，让上层通过接口调用下层，下层不反向依赖上层。
  4. 把稳定业务规则放在核心层，把易变技术细节放在外侧或底层。

具体例子

ProjectPermission 权限模型适合放在 Domain Core 层。 如果未来要新增 OAuth 登录方式，UI 和 API Boundary 可能要改， 但“Owner 不能失去自己 Project 的全部权限”这样的权限规则不应该跟着登录方式变化。 

UI
  -> API Boundary
      -> Application Use Cases
          -> Domain Core
              -> Infrastructure

常见误区

  • 误以为分层就是文件夹分层；真正重要的是依赖方向和职责边界。
  • 误以为层越多越专业；每一层都会增加理解成本，分层要服务于隔离变化。
  • 误让 UI 或脚本直接绕过领域层修改数据；这样会破坏核心规则的一致性。

概念关联

关联概念关系说明
API Boundary / Contracts层与层之间需要清晰接口契约，才能稳定协作。
Domain Core领域核心是分层架构中承载业务规则和不变量的层。
领域不变量分层架构给不变量提供了合适的驻留位置。

精读文章 · 软件工程 / 计算机科学

领域不变量

完整讲解

不变量就是无论系统怎么运行、什么时候执行，都必须一直成立的规则。 在图书馆系统里，“每个读者最多借 10 本书”就是不变量； 在权限系统里，“项目 Owner 必须拥有自己 Project 的全部权限”也可以是不变量。

领域不变量特指业务世界本身固有的规则。今天的权限模型强调 Owner 默认全权限： Project 创建者必须拥有 access、use、delete 等完整权限， 不能被其他用户撤销，也不能被自己放弃。否则 Project 可能变成没人能管理或删除的孤儿资源。

领域不变量应放在 Domain Core 中，而不是只放在数据库约束或 API 校验里。 因为系统入口可能很多：Web API、命令行、后台脚本、迁移任务。只有核心领域模型统一执行规则， 才能让所有入口都绕不过同一条底线。

原理逻辑

  1. 先识别业务中绝不能被破坏的状态或关系。
  2. 把这些规则表达在领域对象、聚合或领域服务中。
  3. 让所有修改入口都必须经过领域核心，而不是直接改数据库。
  4. 用测试覆盖不变量，确保重构和新功能不会偷偷破坏它。

具体例子

权限更新方法可以在每次变更前后检查 Owner 权限。如果操作会让 Owner 失去全部权限， 方法就直接拒绝这次变更： 

function updatePermission(project, actor, change) {
  applyChange(project, actor, change);

  if (!project.owner.hasAllPermissions(project)) {
    throw new Error("Owner must keep full project permissions");
  }
}

这样无论请求来自 API、后台脚本还是迁移工具，只要走到领域核心，就会被同一条规则保护。

常见误区

  • 误以为数据库约束足够；数据库很难表达状态机、权限和业务语义的全部规则。
  • 误以为不变量永远不会调整；它可以随业务版本演进，但修改必须谨慎且成体系。
  • 误把一次接口参数校验当成领域不变量；不变量要覆盖所有入口和所有状态变化。

概念关联

关联概念关系说明
Domain Core领域不变量应放在核心层，成为业务模型的一部分。
ProjectPermission 权限模型Owner 全权限不变量是权限模型的安全底线。
输入契约输入契约守阶段入口，领域不变量守业务状态，都是可执行规则。

概念之间的关联

输入契约、分层架构和领域不变量像一组逐渐深入的边界控制：输入契约在流水线入口确认材料合格； 分层架构在系统内部划清职责和依赖方向；领域不变量在核心层固定业务底线。 它们共同把不确定的协作变成可检查、可审计、可执行的规则。

在系统边界上放稳协议，复杂协作才不会靠默契勉强维持。
