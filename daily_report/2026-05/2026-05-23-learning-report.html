精读文章 · 大语言模型 / 软件工程

提示词工程

完整讲解

提示词工程，简单来说就是“怎么跟 AI 说话才能得到你想要的结果”。如果把大语言模型想象成一个超级博学但有点死板的图书管理员，你跟他说“帮我找本书”和“请帮我找一本 2023 年之后出版的、关于 Python 数据分析的、评价在 4 星以上的中文书”，产生的效果完全不同。前者你会得到一堆反问，后者你能直接拿到精准结果。提示词工程就是把这种“怎么说清楚”的技巧系统化，包含角色设定（“你现在是一个资深程序员”）、任务描述（“请帮我解释这段代码”）、输出格式（“请用要点列表回答”）、边界约束（“不要编造你不确定的函数用法”）以及示例引导（“像这个例子一样处理”）。当这些要素被精心编排成一个提示模板时，模型的行为就会变得可预测、可复用。这里有一个重要的思维转变：提示词不是“一个问题”，而是“一套规范”。就像老师出考试卷子，不仅要有题目，还要写清楚答题要求、字数限制、评分标准。提示词工程就是为 AI 出考卷的学问。

核心要点

  • 提示词工程包含角色设定、任务描述、输出格式、边界约束和示例引导等关键要素，当这些要素被精心编排成提示模板时，模型行为就会变得可预测、可复用
  • 提示词不是“一个问题”，而是“一套规范”——就像老师出考卷，不仅要出题还要写清答题要求和评分标准
  • 提示词工程是系统性地管理模型输入空间的方法，它不是语感，而是一种面向语言模型的程序设计

原理逻辑

  1. 通过角色设定让模型进入特定专业模式，限定回答的知识边界和语气风格，从而缩小输出的不确定性空间
  2. 通过任务描述和输出格式约束，将模糊期待转化为模型能准确执行的文字指令，使输出从“碰运气”变为“可预期”
  3. 通过边界约束和示例引导建立可预测的输出空间，让提示词成为可版本化、可 A/B 测试的程序设计单元

具体例子

想象你开了一家餐厅，雇了一位刚从烹饪学校毕业的厨师。第一天你跟他说“做个菜”，他端上来一盘水煮西兰花。你很失望，但问题出在指令上。第二天你给他详细的菜谱：“做一道宫保鸡丁，用鸡胸肉切丁，花生米先炸过，酱汁比例是生抽两勺、醋一勺、糖半勺，出锅前撒葱花，摆盘时用白盘子，拍照发给我审核。”这次他做出来的菜跟你想象的一模一样。提示词工程就是那张菜谱——你要把脑子里模糊的期待，变成模型能准确执行的文字指令。而且一旦菜谱写成，下次还可以直接用，甚至分享给别的厨师。

常见误区

  • 很多人以为提示词工程就是“说话好听一点”或者“加几个魔法词汇如 think step by step”，但实际上它是系统性管理输入空间的工程方法
  • 提示词工程不是凭语感随意调整措辞，而是需要设计模板时考虑角色一致性、输出边界、错误处理，甚至要进行版本控制和效果对比

Agent 工作流与任务编排

完整讲解

如果你已经理解了提示词工程中“给一个模型配一套精细指令”的思路，那 Agent 工作流就是把多个这样的“模型+指令”组合串联起来，像一个自动化团队一样协同工作。想象你开了一家小型报社。一个人既要去采访、又要写稿、还要排版校对，效率低且容易出错。现实中的报社是分工的：记者负责采集信息，编辑负责整理成文，校对负责检查错误，美编负责排版输出。每个人在自己的岗位上，拿着不同的“任务说明书”（也就是不同的提示词），完成专属的工作，并且把结果交给下一个人。Agent 工作流就是把一个复杂的自动化任务拆解成一系列子任务，每个子任务分配给一个专门的 Agent，按照编排好的顺序执行。Agent 编排的工程价值在于“解耦”和“可检查”。你不需要一个全知全能的超级 Agent，而是可以像搭积木一样，把经过验证的小 Agent 组合成一整条流水线。中间每一步的输出都可以保存下来，出问题时你能回溯到具体哪一步出了问题，而不是面对一团黑箱。

核心要点

  • Agent 工作流把复杂自动化任务拆解为多个子任务，每个子任务分配给专门的 Agent，按编排好的顺序（串行、并行、条件分支）执行
  • 编排的工程价值在于“解耦”和“可检查”——中间每一步的输出都可以保存，出问题时能回溯到具体哪一步，而不是面对一团黑箱
  • Agent 编排的艺术不在于数量多，而在于切分合理——每个子任务要有清晰的输入输出边界，出错时能独立重试，不污染下游

原理逻辑

  1. 将复杂任务按职责拆分为独立的子任务，每个子任务配置专门的提示词和边界约束，让每个 Agent 只负责自己擅长的领域
  2. 通过编排脚本按预定顺序、条件分支或并行方式调用 Agent，形成自动化流水线，每步输出作为下一步的输入，构成可追溯的中间产物链条
  3. 利用解耦和可检查的特性，当某个环节出错时能够独立重试或回滚，而不需要重新执行整个流程

具体例子

有趣的历史类比是印刷术之前的修道院抄经室。如果一位修士要从头到尾独自抄完一本书，速度慢且任何走神都可能导致整页报废。后来的分工方式是：一位修士负责朗读原文，另一位负责抄写，第三位负责校对标点，第四位负责绘制装饰图案。每人只负责自己擅长的一小段工序，出错率大幅下降，产出速度却成倍提升。Agent 工作流就是现代数字版的抄经室分工——每个 Agent 只做一件事，做好后把半成品传给下一个人。当一个环节需要更换人员时（比如换一个更好的校对），你只需要替换那一环，不用重新组织整个流程。

常见误区

  • 很多人认为 Agent 越多越好，用十个 Agent 一定比用一个强。但实际上，每个 Agent 之间的交接都是潜在的出错点和延迟点，盲目增加 Agent 而不设计好契约，效率反而崩溃
  • Agent 编排不是简单地把任务随机拆分，而是需要精心设计每个子任务的输入输出边界，确保交接清晰、可独立验证

概念关联

关联概念关系说明

  2. 多 Agent 之间传递信息时，用自然语言还是结构化 JSON 更好？自然语言灵活但容易歧义，JSON 严格但可能丢失微妙信息，如何取舍？

MCP（模型上下文协议）

完整讲解

MCP 的全称是 Model Context Protocol，即“模型上下文协议”。要理解它，可以先讲一个物理世界的比喻：你家里的电器——台灯、电视机、电脑——都需要用电，但它们不是从电厂直接拉一条专用电线到家里，而是通过墙上统一的插座来获取电力。国家标准插座规定了两孔还是三孔、零线火线的顺序、最大电流承载。只要电器的插头符合这个标准，插上去就能工作。这极大地简化了电器和电网之间的连接。在大语言模型的世界里，MCP 就是那个“插座标准”。一个 LLM 在工作时经常需要外部帮助：查数据库（资源 resources）、调用计算器或搜索引擎（工具 tools）、使用预置对话模板来调整角色（提示 prompts）。在没有 MCP 之前，每次开发者想给模型加一种新能力，都要自己写专门的对接代码——相当于每买个新电器就自己拉一根电线。而 MCP 定义了三种标准化的接口，不管 LLM 本身怎么换、怎么升级，只要它支持 MCP，就能无缝接入预配好的工具和资源——让 LLM 应用开发从手工接线走向即插即用。

核心要点

  • MCP 定义了三种标准化的“插孔”：工具（tools）封装特定操作接口、资源（resources）提供可读数据源、提示（prompts）注册预配置的提示模板
  • MCP 不改变模型本身任何参数，只是一套通信协议——就像 USB 协议不会让电脑变快，但标准化了设备接入方式
  • 通过 MCP，LLM 应用开发从“每买一个电器就自己拉一根电线”变成“统一插座即插即用”，极大地提升了工程效率和解耦程度

原理逻辑

  1. MCP 将模型外部能力抽象为三种标准化原语：tools 封装操作接口让模型可以调用外部功能、resources 提供模型可读取但不可随意修改的数据源、prompts 注册预配置的提示模板供客户端按需选取
  2. 所有支持 MCP 的 LLM 客户端通过统一协议访问这些能力，实现模型与外部资源的彻底解耦——想换模型直接换，因为接口不变；想给工具加参数只改工具定义的 schema，不影响其他模块
  3. 协议化的接口契约让工具、提示、资源可以独立迭代、独立测试、独立部署，体现了服务端模块化与关注点分离的软件工程原则

具体例子

想象你在 20 世纪初买了一台新发明的电风扇，但发现它的插头是三角形的，而你家里的插座都是圆形的。你只有两个选择：要么重新装修布插座，要么找人把电风扇的插头剪掉换一个圆形的。如果买回来的每一件电器插头形状都不同，那家里就是个插座万花筒，维修和使用都是噩梦。这就是大语言模型应用开发在 MCP 出现之前的状态：每个工具、每个数据源、每个提示模板的接入方式都是定制的。MCP 就如同国家出台了统一的插座标准，从此家电厂按标准生产插头，建筑商按标准安装插座，你买回任何电器直接插上就能用。

常见误区

  • 很多人以为 MCP 是一种“能让 LLM 变聪明的新技术”或“AI 的新模型”。实际上，MCP 不改变模型本身任何参数，不提升模型的语言理解能力，它只是一套通信协议
  • MCP 的价值在于工程效率和解耦，而不在于模型能力的魔法飞跃——就像 USB 协议不会让你的电脑变快，但标准化了键盘、鼠标、U 盘的接入方式

继续深入问题

  2. 如果多个不同的 LLM 应用同时通过 MCP 访问同一组 resources，如何管理并发读取、数据一致性和资源配额，避免某个应用占满连接池？

概念之间的关联

这三个概念共同描绘了一条从“人指导模型”到“系统自我组织”的工程化路径。提示词工程就像给模型写一份详细的工作说明书，Agent 工作流像是把不同工种的工人按照流水线组织起来，而 MCP 则像是为这条流水线配置了标准化的工具箱和原料仓库。它们不是孤立的知识点，而是一条从微观到宏观、从手工到自动化的工程升级链。理解它们之间的递进关系，就能理解为什么让模型可靠工作需要在语言、流程和协议三个层面同时发力。

今日最值得记住的一句话

提示词工程是 LLM 时代的“程序设计”：通过精心编写提示模板，我们可以稳定地控制模型的输出风格、格式和逻辑边界，而不仅仅是靠运气获得好结果。
