声明式视觉契约：CSS 作为约束系统

完整讲解

想象你要装修一间房子。有一种做法是，你给装修队发一份详细的指令清单：把沙发搬到离东墙 3 米的位置，把台灯放在茶几右上角 20 厘米处，窗帘用米黄色。这叫做“命令式”——你要一步步告诉执行者怎么做。另一种做法是，你画好一份设计图纸，上面标注了各个区域是干什么的、家具之间要保持多少距离、整体的配色方案——装修队看到图纸，就知道该怎么摆放能符合你的期望。这叫做“声明式”。 CSS 样式表就是网页的“设计图纸”——一种声明式语言。它不是告诉浏览器“先画一个红框，再把它往左边挪 5 像素”，而是声明规则：“所有标题字体应该用深蓝色、居中显示”，“表格的每一行之间留出 12 像素间距”，“当屏幕宽度小于 600 像素时，侧边栏自动收起来”。浏览器读到这些规则后，自己负责计算具体怎么布局。 今天的材料中，一个学习报告系统的 CSS 文件被大幅修改——这不是简单的换颜色，而是因为报告展示的需求变了，那份“视觉契约”需要升级。当契约升级时，生成 HTML 的脚本（施工队）和检查输出是否合规的校验器（验收员）都要同步更新，否则就会出现“按新图纸施工但按旧标准验收”的矛盾局面。这就是为什么一个小小的样式修改，会联动触发生成流水线和验证规则的同步调整。

原理逻辑

  1. 浏览器引擎读取 CSS 规则后，自动计算元素在二维空间及 z 轴上的位置、尺寸和样式，无需手动干预
  2. 样式表将视觉规则集中声明，形成对页面所有元素呈现方式的统一控制
  3. 媒体查询允许根据设备特性（如宽度、分辨率）声明不同的布局规则，实现响应式设计

具体例子

在 DailyLearningAssistant 项目中，通过增加 .chart-container 的 max-width 和响应式媒体查询规则，开发人员无须编写 JavaScript，就实现了图表容器在大屏居中、小屏自适应宽度的效果。同时，生成报告 HTML 的脚本也需要保证输出的结构符合新样式表的选择器约定，而校验器也要更新检查规则，确保生成的 HTML 里确实包含样式表所期待的那些 CSS 类名。

常见误区

  • 以为 CSS 只是给网页“化妆”——调整颜色和间距，而忽略了它是布局系统的约束语言
  • 修改样式后，忘记同步更新依赖该样式的生成脚本和校验规则，导致显示与验证脱节

话题-会话的父子关系建模

完整讲解

你整理过电脑桌面上的文件夹吗？最开始你可能把所有文件都摊在桌面上——一个长长的列表，找什么都靠肉眼扫描。当文件数量超过 20 个，你就开始慌了。于是你创建几个文件夹：“学习资料”“照片”“工作项目”，然后把这些文件分门别类拖进去。突然，世界清净了——你的 50 个文件现在只需要看 3 个文件夹图标，想要什么点进去就能找到。你刚才做的事情，在软件开发上就叫“引入层级容器”——用分类容器把原本扁平化的元素组织成树形结构。 今天 pi-web 项目做的就是这样一件事。这个项目是一个会话管理界面，原来所有的对话记录以平面列表形式展示在侧边栏——就像摊满桌面的论文草稿。开发人员新增了“话题”这个概念，让一个话题可以包含多个会话。这不仅仅是界面上的小改动，而是整个数据模型的重新定义：原来会话是最高层级的“公民”，现在变成话题的“子民”。 这种父子关系的建模会影响系统的方方面面。API 路由从 /api/sessions 变成了 /api/topics/3/sessions，表示“先找到第 3 号话题，再看它下面的会话”。前端组件的状态管理从一维列表变成了树形结构，需要处理展开/折叠、父级选中时的子级状态等新问题。这个改动虽然代码量不大，但它是从“平面思维”到“层级思维”的一次重要设计决策。

原理逻辑

  1. 数据模型定义显式的归属关系，API 路径通过嵌套资源（如 /topics/3/sessions）体现层级
  2. 前端组件从扁平列表渲染升级为递归的树形渲染，状态管理从一维扩展到二维（展开/折叠、选中态传递）
  3. 层级建模本质是划定聚合边界，明确变化一致性的范围，与领域驱动设计中的聚合思想相通

具体例子

在 pi-web 项目中，引入话题容器前，API 返回扁平的会话列表；引入后，返回嵌套结构：话题包含多个会话。前端侧边栏组件 SessionSidebar.tsx 需要从遍历数组重构为递归渲染，先渲染话题节点，再渲染其下的会话子节点。同时，折叠状态从简单的“哪个会话被选中”转变为“哪些话题展开了、展开的话题下哪个会话被选中”的二维状态管理。

常见误区

  • 以为引入父子关系只是在数据库里添加一个外键字段，而忽略了 API 设计、类型定义、前端组件和状态管理的全面重构需求
  • 没有考虑深层嵌套导致的性能问题（如递归渲染大量节点），或未能提前规划懒加载策略

概念关联

关联概念关系说明

模块注册的显式化与安全删除范式

完整讲解

你用过图书馆的借书系统吗？每本书入库时，管理员会同时做两件事：把书摆上对应书架，并在图书管理系统里登记这本书的信息（书名、位置、状态）。这两步缺一不可——如果只上书架不登记，读者永远搜不到这本书；如果只登记不上书架，读者按索引找到位置却拿不到实体书。反过来，当一本书需要下架时，管理员必须同时完成两个反向操作：从书架上取走实体书，并在系统里注销记录。如果只拿走书忘了注销，系统会认为书还在，下次读者跑到那个位置就会扑空。 在软件开发中，一个“模块”就像一本上架的书。模块通常由三部分组成：实际干活的业务逻辑代码（书本身）、定义接口的类型/模式文件（书的简介和规格）、以及在注册文件中的登记条目（图书馆系统中的记录）。注册文件是整个系统知道“有哪些可用模块”的唯一入口。当开发者需要删除一个模块时，必须遵循完整的注销流程：删除业务逻辑文件、删除类型定义文件、从注册文件中移除引用条目。如果漏掉任何一步，系统在运行时可能试图寻找一个已经不存在的模块，导致崩溃或不可预期的错误。 今天 mcp 仓库的一个提交展示了这个范式的实际操作：删除一个工具时，同时移除了它的业务逻辑文件、模式定义文件和注册文件中的引用。这三步操作的协同，体现了“显式依赖管理”的核心原则：模块的存在与消失都应当在系统中留下清晰的痕迹。

原理逻辑

  1. 模块的生命周期由注册文件显式管理，任何模块的增减都必须在注册文件中同步更新
  2. 删除流程遵循“先解除引用、再移除实现”的顺序，确保依赖图完整
  3. 悬空引用（dangling reference）在静态语言中会导致编译错误，在动态语言中可能导致运行时崩溃，显式注销可以彻底杜绝这类问题

具体例子

在 mcp 仓库中，要删除一个叫 weather-tool 的工具，正确的操作包括：第一步，删除业务逻辑文件 src/tools/weather-tool.ts；第二步，删除模式定义文件 src/schemas/weather-tool-schema.ts；第三步，在注册文件 tools/index.ts 中移除导入和 registerTool('weather', weatherTool) 条目；第四步，运行测试和构建，确认无其他文件引用。这四步操作确保不留下任何幽灵引用。

常见误区

  • 删除时只移除主代码文件，认为类型定义和注册引用会“自动失效”，结果在启动时因注册表仍指向已删除文件而崩溃
  • 忽视注册文件的版本控制，导致合并冲突或多人协作时遗漏注销步骤

概念关联

关联概念关系说明
