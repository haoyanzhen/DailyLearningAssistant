工作台分层与流程入口收口

这一天的核心推进是把 Project Workspace、Construction Workspace、Construction Run、 Research Session 和 Review Run 的边界理顺，让长期配置、一次执行、开放会话和流程式写作各自回到合适的位置。

统计窗口 2026-05-02
提交数量 4 次文档提交
核心对象 Workspace / Run / Session
主要收益 分层清晰，入口更轻

今日知识点概览

5 月 2 日的提交共同完成了一轮信息架构收敛：流程式 Agent 有了统一步骤容器， Construction Workspace 被定位为长期配置容器，Construction Run 回到一次执行记录的位置， Project Workspace 的顶层入口也因此更清楚。

步骤容器 Construction Run 与 Review Run 共享步骤栏、当前阶段区和结果查看结构。
构建层级 长期配置留在 Construction Workspace，单次执行沉淀为 Construction Run。
入口收口 历史 Construction Run 不再挤在项目顶层，而是在对应工作台内部查看。
步骤容器 流程式 Agent

先把流程页面骨架补上

当天第一条线索是 step page manager：构建和综述这两类流程式 Agent 不再依赖零散页面拼接， 而是使用统一步骤页承载状态、确认点、失败恢复和阶段性结果。

Construction Workspace 长期配置

把配置对象和执行对象拆开

每个 Project 保留一个长期存在的 Construction Workspace，用于保存检索词、自动更新开关和数据源策略； 每次手动或自动构建再创建 Construction Run，作为执行快照与历史记录。

Project Workspace 上下文边界

重新收紧项目工作台展示范围

系统级入口、Project 内部工作台、信息面板、对象切换和上下文恢复被重新分层。 Research Session 保持开放式会话，Review Run 保持流程式写作运行，命名逐渐贴近交互形态。

信息架构 入口修剪

把历史执行入口放回相关工作台

Project Workspace 总面板只展示顶层工作对象。历史 Construction Run 进入 Construction Workspace 后再查看，用户先选择工作台，再查看该工作台内部历史，导航负担更低。

今日最值得记住的一句话

长期配置对象和一次执行对象必须拆开；入口应该跟随用户真正的工作上下文，而不是跟随系统内部对象数量膨胀。
