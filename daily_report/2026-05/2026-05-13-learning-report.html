先把边界、流程和运行状态讲清楚

今天的学习主线聚焦 AI 参与复杂工程之后的秩序感：先用契约式开发规定系统边界， 再用渐进披露组织复杂内容，最后用状态机与并发锁保护长任务和多 Agent 协作的运行态。

今日知识点概览

今天的三个知识点都在回答同一个问题：复杂系统怎样不被“看起来很聪明”的自动化带偏。 契约式开发让 AI 和人都知道边界，渐进披露让复杂知识按深度被读到，状态机与并发锁让运行中的系统不会互相踩踏。

契约式开发 在实现前明确产品目标、API、数据、权限、错误路径和验收标准。
渐进披露 把同一份复杂内容分成简读和精读，让用户先抓主线再进入细节。
状态机与并发锁 用状态迁移和关键资源保护，避免长任务、多窗口、多 Agent 操作互相覆盖。

今日知识主线

当 AI 参与复杂工程时，真正重要的不是让它多写代码，而是先把输入输出、展示层次、 状态变化、权限边界和验收条件讲清楚。自动化越强，越需要可审核、可复盘、可理解的轨道。

软件工程 ★★★★

契约式开发

Contract-based Development

先把系统边界、输入输出、权限规则和验收标准讲清楚，再让实现服从这些约定。

核心要点

  • 契约不只是 API 文档，也包括业务规则、数据约束、错误路径和人工审核边界。
  • AI 很会补细节，边界不清时也很容易自作主张。
  • 好契约固定关键边界，同时给具体实现留下合理空间。

小例子

“上传 PDF 并补全元数据”要说明登录要求、解析顺序、外部数据源失败策略、LLM 草稿是否需要用户确认。

记忆句：先签清楚接口合同，再让代码开始奔跑。
人机交互 ★★★

渐进披露

Progressive Disclosure

不一次性塞满所有信息，而是按注意力、任务阶段和理解深度逐步呈现内容。

核心要点

  • 渐进披露不是隐藏信息，而是安排信息出场顺序。
  • 简读负责判断和回忆，精读负责完整学习。
  • 入口必须清楚，过度折叠会让页面变成寻找答案的负担。

小例子

学习日报先展示今日主线和一句话解释，点击精读后再展开原理、例子、误区和开放问题。

记忆句：好的界面不是少给信息，而是让信息按时出现。
后端工程 ★★★★★

状态机与并发锁

State Machine and Concurrency Lock

状态机规定系统能从哪里走到哪里，并发锁保证关键资源不会被多个操作同时改乱。

核心要点

  • 状态字段本身不够，真正重要的是状态迁移规则。
  • 前端禁用按钮只能改善体验，不能替代后端或数据层的并发保护。
  • 锁要缩小范围、缩短持有时间，并明确抢不到锁时的失败策略。

小例子

同一个 Project 最多只能有一个 active Run，开始运行前要先获取 Project 级锁。

记忆句：状态机管路线，锁管同一时间谁能过门。

精读文章 · 软件工程

契约式开发

完整讲解

可以先从做饭理解契约。你和朋友说“你买菜，我炒菜”，这只是粗分工。朋友可能买十斤土豆回来， 你却准备做清蒸鱼；你以为他会买葱姜蒜，他以为调料家里都有。问题不一定是没人努力，而是约定太模糊。

契约式协作会提前说明今天做什么菜、需要哪些食材、每种食材多少、最晚几点买回来、买不到鱼用什么替代、 谁负责确认最终菜单。放到软件工程里，契约式开发就是在正式实现之前，先把产品目标、架构边界、API 输入输出、 数据模型、权限规则、错误处理和验收标准写成明确约定。代码实现必须服从这些约定。

在 AI 辅助开发里，契约尤其重要。AI 很擅长补全细节，但边界不清时也容易根据常见模式自行补字段、补权限、 补流程，甚至把安全逻辑写得过于宽松。所以 Product Brief、Architecture Note、Data Model、API Contract、 Vertical Slice 和人工审核边界，本质上都在做同一件事：先建立契约，再让 AI 在契约内工作。

原理逻辑

  1. 边界清晰：功能需求描述用户能力和业务规则，不应过早混入数据库字段名和展示层细节。
  2. 责任清晰：前端负责展示交互，后端负责业务规则，数据库负责持久化约束，AI 生成候选实现，人类审查高风险部分。
  3. 验证清晰：契约要能被检查，例如禁用账号会话失效、密钥不得明文展示、同一 Project 不能同时创建两个 active Run。

具体例子

如果要做“上传 PDF 论文并补全元数据”，模糊需求可能只有一句“用户上传 PDF，系统自动识别论文信息”。 契约式开发会写清楚：用户必须登录；PDF 上传后先本地解析标题、作者、年份；不完整时再查外部数据源； 仍不完整时可调用 LLM 生成候选草稿；LLM 草稿不得直接覆盖正式元数据，必须由用户确认； 远程 PDF 访问必须使用鉴权或签名 URL；解析失败时保留原始文件并返回可理解的错误状态。

常见误区

  • 契约不是只有 API 文档。业务规则、权限边界、错误路径、数据约束、人工审核范围和验收标准也属于契约。
  • 契约不是越细越好。好契约固定关键边界，但不把每个函数名、每个 UI 细节都提前锁死。
  • 用了 AI 不是可以少写契约，而是更要写清楚契约。强模型会主动补全内容，边界不清就容易偏离真实需求。

概念关联

关联概念关系说明
API 契约规定系统之间如何通信，是契约式开发的重要落点。
Schema 分层区分展示数据、接口数据和持久化数据，避免层次混乱。
AI 决策边界说明哪些实现可交给 AI，哪些安全、权限、状态逻辑必须人工重点审核。
垂直切片开发用一条完整功能链验证契约是否真的闭合。

精读文章 · 人机交互

渐进披露

完整讲解

想象你打开一份体检报告。如果第一页直接塞满几十项指标、医学缩写、参考区间和检查说明，很容易马上头大。 如果第一页先告诉你总体是否正常、哪几项需要关注、是否需要复查、想看细节可以展开每个指标，理解成本就会低很多。

渐进披露指的是：不要一次性把所有信息、功能和选项全部展示给用户，而是按照用户的注意力、任务阶段和理解深度， 逐步呈现更详细的内容。它不是隐藏信息，而是安排信息出场顺序。好的渐进披露像一位老师，先告诉你结论， 再慢慢展开推导、例子和拓展阅读。

学习日报里的“简读/精读”就是典型应用。简读适合快速回顾：今天讲了什么、主线是什么、结论是什么； 精读适合深入学习：完整讲解、例子、误区、关联概念和开放问题。同一份内容服务两类阅读状态， 用户不必在“太浅”和“太累”之间二选一。

原理逻辑

  1. 降低认知负担：人的注意力有限，先展示主线和关键行动可以帮助用户建立心理地图。
  2. 匹配任务阶段：刚打开日报时用户需要判断是否值得读，决定深入后才需要完整解释和更多例子。
  3. 保护高级功能：基础功能直接可见，高级能力在需要时展开，既照顾新手也照顾熟练用户。

具体例子

如果今天日报有契约式开发、渐进披露、状态机与并发锁三个概念，不使用渐进披露时，页面可能一口气展示几千字。 使用渐进披露后，可以先显示今日主线和三条一句话结论；每个概念先显示领域、难度和核心解释； 点击“精读”后再展开完整原理、例子、误区和开放问题。这样既不牺牲完整性，也不强迫每位读者每次都从头精读。

常见误区

  • 渐进披露不是把内容藏起来。隐藏会让用户找不到，渐进披露要让用户在合适时机找到。
  • 折叠不是越多越清爽。常用信息应该直接可见，低频细节才适合收起。
  • 摘要不能替代正文。简读负责帮助判断和回忆，精读负责完整学习。

关联概念关系说明

精读文章 · 后端工程

状态机与并发锁

完整讲解

状态机可以理解成一张路线图。学生作业可能有未提交、已提交、批改中、需要修改、已完成这些状态。 它们之间不能随便跳：未提交可以变成已提交，但不应该直接变成已完成；批改中可以变成需要修改或已完成， 但不该突然回到未提交，除非有特殊规则。

状态机就是用明确的状态和迁移规则描述系统行为的方法。它回答：系统可能处于哪些状态； 每个状态下允许发生哪些动作；动作发生后状态会变成什么。AI Agent 状态机、Run 状态、取消、暂停、 恢复、重试和只读状态，都属于这个问题。

并发锁处理的是另一类风险：多个事情同时发生。两个用户几乎同时点击“开始构建”，或者两个自动化任务同时处理同一个 Project， 如果没有保护，就可能创建两个 active Run，两个任务都以为自己是唯一正在运行的任务，最后互相覆盖结果。 并发锁要求关键资源正在被一个操作使用时，其他操作必须等待、失败或进入队列。

原理逻辑

  1. 状态机规定这个 Run 现在是什么状态，以及当前状态下能做什么动作。
  2. 并发锁保证同一时间不会有两个操作同时把状态改乱。
  3. 二者结合后，可以在进入 running 的瞬间获取 Project 级锁，抢不到就拒绝创建新的 Run。

具体例子

一个 Project 自动构建知识库时，可以设计如下状态：

状态含义允许动作
idle当前没有运行任务start
queued已排队等待执行cancel
running正在执行pause、cancel
paused已暂停resume、cancel
failed执行失败retry、archive
completed执行完成archive、start_new
cancelled用户取消retry、archive

用户点击 start 时，后端先尝试获取 Project 级锁。如果锁获取成功，并且当前没有 active Run， 才创建新 Run 并把状态改为 queued 或 running。锁获取失败时，系统返回“该 Project 已有任务正在启动或运行”， 而不是偷偷创建第二个任务。

常见误区

  • 前端按钮禁用不能解决并发。用户可能开两个窗口，请求也可能来自 API、脚本或重试机制。
  • 状态字段不是随便加几个字符串就够。真正重要的是哪些状态能转到哪些状态，谁有权限触发，失败时如何处理。
  • 锁不是单纯让系统变慢。没有锁可能导致数据损坏，工程上要缩小锁范围、缩短持有时间、明确失败策略。

概念关联

关联概念关系说明
MVP 范围控制首版可以减少复杂状态，但不能省略关键状态边界。
验收标准设计并发保护和状态迁移应写成可测试的验收条件。
内容修改保护防止多个 Agent 或流程同时覆盖人工内容。
多阶段自动化流水线流水线运行时需要状态记录、失败恢复和重跑规则。

今日最值得记住的一句话

高质量 AI 协作不是放任模型自由生成，而是把边界、层次、状态和验收条件设计清楚。
