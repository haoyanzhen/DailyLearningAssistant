版本控制边界与中间产物管理

完整讲解

版本控制工具（比如 Git）就像一个项目的记忆相册，记录着每一次有意义的变化。当我们开始用自动化流水线每天生成一堆报告、总结、中间分析文件时，就面临一个选择：这些自动生成的文件，要不要也存进相册？如果不加区分全部存进去，仓库就会快速臃肿，而且每天的微小变化会产生大量无意义的提交记录，把真正重要的代码改动淹没掉。版本控制边界的思考，就是帮我们划一条线：哪些文件是“源知识”——比如源代码、提示词、架构设计文档——它们是你思想的原始表达，值得永久珍藏；哪些文件是“运行产物”——比如每天的自动总结——它们只是流水线在特定输入下的一次输出，只要源代码和输入还在，随时可以重新生成。这个边界一旦确立，我们就可以把运行产物排除出版本跟踪，甚至删除历史痕迹，让仓库轻装上阵，只保留真正体现设计演进的内容。这在软件工程里，是保证项目长期可维护的重要治理手段。

核心要点

  • 区分“源知识”（源代码、提示词、设计文档）与“运行产物”（每日自动生成的中间报告）
  • 将可自动重建的运行产物排除出版本跟踪，让仓库轻装上阵，只保留体现设计演进的内容
  • 只有流水线具备可重复构建能力时，取消跟踪中间产物才是安全的

原理逻辑

  1. 版本控制就像项目的记忆相册，应只记录有意义的变化——源代码、设计文档等“源知识”的演变
  2. 自动化流水线每天生成的中间文件属于“运行产物”，是特定输入下的输出，可随时从源重建，不应污染仓库历史
  3. 确立版本控制边界后，通过 .gitignore 将运行产物排除跟踪，仓库只保留设计知识，实现轻量化和长期可维护

具体例子

在当天提交中，开发者批量删除了 prework 目录下大量历史总结文件，并修改 .gitignore 将它们排除在外。但 design 目录下的架构文档、状态设计文档却继续留在仓库里。因为前者是每一次运行产出的临时报告，后者是记录“为什么这样设计”的源知识。

常见误区

  • 误以为“只要生成过一次，就该永久保存以便回溯”。但回溯的真正对象应是“生成逻辑”而非“生成结果”。如果流水线不可重复，留着结果就是一堆死数据；如果可重复，强留中间产物只会加重仓库负担、混淆版本历史。
  • 当流水线还在手工作坊阶段，每个中间文件都需要人工检查修改，纳入Git是合理的。但当流水线成熟到可以稳定复现时，这些中间产物就退化为“缓存”，取消跟踪是工程演进的标志。

流水线状态管理与生命周期

完整讲解

想象你在一个复杂的组装车间，经过多个工位才能完成一件产品。如果没有一个记录卡片跟随产品走，没人知道它现在到了哪个工位、该做什么、上一个工位有没有做好。自动化流水线也一样：当多个智能体按顺序执行时，系统必须有记忆——记住当前跑到哪一步了，这一步的输入是什么，输出是什么，下一步该做什么，如果某一步出错该怎么处理。这就是“流水线状态管理”：用一个结构化的状态对象，记录一次运行过程的“生命状态”。而“生命周期”则把这个过程划分成明确的阶段，比如初始化、正在执行某个步骤、全部完成、或失败终止，并规定阶段之间允许怎样跳转。这背后的计算模型是有限状态机，一种经典的计算抽象。这样一来，流水线不再是一团难以追踪的函数调用，而是一个状态清晰可见、行为可预测的系统。就算你在半夜收到告警，一看状态记录就知道卡在哪一步。

核心要点

  • 用一个结构化的状态对象记录一次运行过程的“生命状态”——当前阶段、输入输出和历史
  • 生命周期把流水线划分成明确的阶段（初始化、执行中、完成、失败），并规定阶段之间的转移规则
  • 状态持久化让故障排查和恢复变得可能，即使半夜收到告警，一看状态记录就知道卡在哪一步

原理逻辑

  1. 状态对象记录当前阶段、输入输出和历史，相当于给流水线装上了“运行时记忆”
  2. 生命周期将执行过程划分为明确的阶段（初始化、执行中、完成、失败），并用有限状态机规定阶段间的合法跳转
  3. 状态持久化存储后，流水线不再是黑盒的函数调用链，而是可追踪、可恢复的透明系统

具体例子

当天仓库新增的 orchestrator/state.py 和 lifecycle.py 就是这一概念的落地。假设流水线有四个步骤：汇总工作、分析概念、撰写解释、发布报告。状态对象记录结构如：{ 'current_stage': 'concept_analysis', 'stage_history': [{ 'stage': 'work_summary', 'status': 'success', 'output': 'summary.json' }], 'error': null }。生命周期定义则确保在“撰写解释”之前必须已完成“概念分析”，否则状态机拒绝跳转。

常见误区

  • 很多人以为只要把函数按顺序写在一个脚本里，执行过程就是天然的“状态管理”。但脚本顺序只规定了静态的调用关系，一旦执行中途出现异常、被手动重启、或者需要跳过某一步，没有显式状态记录的脚本就会让人抓瞎：你不知道之前到底执行到了哪里，也不知道重启后该从哪里继续。
  • 显式状态管理的价值恰恰在于把“运行时记忆”持久化，让故障排查和恢复变得可能，而不是依赖脚本的自然执行顺序。

大语言模型调用与输出约束

完整讲解

大语言模型就像一个知识渊博但偶尔随性的朋友：你请他帮忙写一封邮件，他可能写得文采飞扬，也可能临时发挥起来，加些你不需要的内容，甚至把格式弄乱。在一条自动化流水线里，你不能容忍这种随意——因为下一个步骤的代码期望接收到的，是一个结构精准、字段完好、没有多余信息的 JSON 对象，而不是一段自由散漫的散文。因此，在把 LLM 接入工程系统时，必须在它的输出后面加上一道“安检”：先用统一的接口去调用模型（处理通信、超时、重试），然后对模型的回答进行结构化校验——比如检查它是不是合法的 JSON，有没有缺少“概念名称”这个必填字段，有没有多出不该有的内容。如果校验不通过，就触发重试或降级逻辑。这就是“LLM 调用与输出约束”的核心：用一层代码护盾，把模型概率性的自然语言输出，转化为下游代码能够安全消费的确定性数据。提示词负责“软约束”，告诉模型应该输出什么；校验层负责“硬约束”，检查模型是不是真的照做了。

核心要点

  • 用统一调用接口处理与大语言模型的通信、超时和重试，避免各Agent各自实现的不一致
  • 对LLM回答进行结构化校验：检查是否为合法JSON、是否包含必填字段、是否有多余内容
  • 提示词负责“软约束”（告诉模型应输出什么），校验层负责“硬约束”（检查模型是否照做），两者结合才是工程上可靠的设计

原理逻辑

  1. 大语言模型的本质是基于概率的文本续写，即使提示词写得再好，也可能因上下文噪声或随机采样而产生格式偏差
  2. 软件工程中不能依赖“一般会好”，必须用统一调用接口管理通信和超时，再用校验层对输出进行结构化检查
  3. 提示词是“软约束”（定义输出规格），校验层是“硬约束”（保证规格被遵守），两者构成分层防御体系

具体例子

假设流水线中有一个 Agent 负责从当日工作总结中提取概念。它的提示词要求模型输出一个 JSON 数组，每个元素包含 concept、domain、explanation 字段。模型返回结果后，validators.py 会检查：这是不是合法的 JSON？是不是数组？每个元素有没有包含这三个字段？如果某个对象缺少 explanation，校验失败，系统会记录错误并决定是重试还是进入降级处理。

常见误区

  • 很多人误以为只要提示词写得足够好，模型就会百分百按照格式输出。但模型的本质是基于概率的文本续写，它可能因为上下文噪声、罕见输入或随机采样而产生格式偏差。提示词只是提高了合规的概率，校验层才是确保合规的底线。
  • 软件工程中不能依赖“一般会好”，而必须用代码强制校验。提示词规定了模型“应该”输出什么，校验层则确认它“确实”输出了这些，两者缺一不可。

概念之间的关联

这三个概念共用同一条逻辑纽带：从“运行一次”到“运行一百次”的可靠性升级。版本控制边界的确立，正是因为流水线已进入“可重复构建”阶段——中间产物可通过源代码和提示词自动重建，不必再占用Git历史。而实现这种自信，必须依赖流水线状态管理：每次运行被记录在状态对象中，阶段转移有明确规则。与此同时，大语言模型的输出充满不确定性，系统引入统一的调用入口与输出校验层，把概率性文本约束成确定的JSON结构。状态机的确定性与输出校验的确定性，共同支撑起流水线的可复现性；可复现性又反过来让版本控制边界的收缩变得安全且合理。

今日最值得记住的一句话

真正需要存档的是能让结果被重新构建的一切——输入数据、代码、配置，而不是结果自身。
